## 0. 최종 변경 요약

### 0.1 이번 업데이트의 핵심 판단

이번 P2 인력 운영은 기존처럼 여러 엑셀 시트를 각각 별도 화면으로 옮기지 않는다.

엑셀 시트의 성격을 다음 4가지로 구분한다.

1. **마스터 데이터**
   - 관리자 화면에서 등록·수정·관리한다.
   - 예: 인력재직현황, 역할과직무.

2. **운영 원장 데이터**
   - 실제 인력 배정의 기준 데이터다.
   - 별도 사이드바 메뉴보다는 운영 화면 또는 상세 화면의 데이터 소스로 사용한다.
   - 예: 인력_투입현황MM.

3. **운영 스냅샷/현황판**
   - 본부장/팀장/운영자가 자주 보는 화면이다.
   - MVP에서는 `인력 투입 현황`을 메인 화면으로 노출하고, 상태 KPI 카드 필터로 대기/제안 인력 목록을 확인한다.
   - 예: 인원별_투입(현재), 대기현황, Ⅱ.3.대기·제안인원.

4. **보고/집계 데이터**
   - 별도 입력 화면이 아니라 백엔드 집계/API 또는 KPI/보고 화면의 데이터 소스로 둔다.
   - 예: 월별투입현황, Ⅱ.1.주간현황, Ⅱ.2.월별가동현황.

### 0.3 기존 DTL 대비 주요 변경

| 항목 | 기존 1차 업데이트 | 이번 2차 업데이트 |
|---|---|---|
| 인력 메뉴 | 인력배치/투입현황 1개만 노출 | `인력 투입 현황` 1개 노출. `대기/제안 현황` 별도 메뉴는 구현하지 않음 |
| 대기현황 | 주간현황 하단 통합 | MVP에서는 인력 투입 현황의 대기 상태 필터로 대체 |
| 대기/제안인원 | 주간현황 하단 통합 | MVP에서는 인력 투입 현황의 제안/수행 상태 필터로 대체 |
| 인력_투입현황MM | 인력배치/투입현황 내부 목록 | 숨김 상세 라우트/배정 원장 데이터로 사용 |
| 인력 마스터 | 관리자 영역 이동 | 관리자 > 인력 관리에서 우선 구현 권장 |
| 월별투입현황 | 월별 MM 요약으로 사용 | 프론트 계산 금지, 백엔드 집계/API 기준 |
| 제안PRJ/이행PRJ | 보류 또는 보고 영역 | 별도 페이지 삭제, 업무수행현황 필터와 화면별 엑셀 다운로드로 대체 |

### 0.4 3차 업데이트 확정 사항

| 항목 | 확정 방향 |
|---|---|
| 현재 메뉴 구조 | DTL 기준으로 수정한다. 인력 메뉴는 `인력 투입 현황`만 노출하고, `대기/제안 현황` 별도 메뉴는 구현하지 않는다. |
| 인력 관리 라우트 | 기존 `/people/employment`를 유지하되 `관리 > 인력 관리` 메뉴에만 노출한다. |
| 인력 투입 현황 라우트 | `/people/current-assignments`로 구현한다. |
| 대기/제안 현황 라우트 | 별도 라우트를 구현하지 않는다. 대기/제안/수행 목록은 `주간현황` 테이블과 `인력 투입 현황` 상태 필터로 확인한다. |
| 인력 화면/API 현황 | 현재 mock 또는 P1성 구현은 참고만 한다. 본 문서는 P2 신규 구현 기준이다. |
| 역할/직무 기준 | `roles` 신규 테이블/API로 구현한다. 문자열/enum 임시 처리는 MVP 기준에서 제외한다. |
| 배정 취소 상태 | `AssignmentStatus.cancelled`를 추가한다. 취소된 배정은 삭제하지 않고 이력으로 보존한다. |
| 근무 위치 구분 | `WorkLocationType` enum을 추가한다. 값은 `onsite`, `remote`, `hybrid`로 한다. |
| 주간 KPI | `GET /api/weekly-kpi-summaries` API는 제공하되, MVP에서는 별도 테이블 없이 `current_assignment_snapshots`, `weekly_load_snapshots` 기반 파생 응답으로 구현한다. |

## 1. 최종 메뉴 구조

### 1.1 전체 메뉴

```text
홈
└─ 홈

프로젝트
├─ 업무수행현황
└─ 진행이력

숨김 라우트
└─ 프로젝트 상세
   ├─ 업무수행현황에서 프로젝트 클릭 시 진입
   └─ 사이드바에는 노출하지 않음

인력
└─ 인력 투입 현황

숨김 라우트
└─ 인력 상세
   ├─ 인력 투입 현황에서 인력 클릭 시 진입
   └─ 사이드바에는 노출하지 않음

KPI/보고
├─ 주간현황
└─ 월별가동현황

관리
├─ 사용자/권한 관리
├─ 기준정보 관리
├─ 프로젝트 관리
├─ 인력 관리 (/people/employment)
├─ 공휴일 관리
└─ 월마감/스냅샷
```

### 1.2 인력 영역 메뉴

```text
인력
└─ 인력 투입 현황
```

인력 영역은 운영자가 실제로 매일 보는 화면만 둔다.

- `인력 투입 현황`: 기준일 현재 누가 어디에 투입되어 있는지 확인하는 메인 운영 화면
- `대기/제안 현황`: 별도 메뉴로 구현하지 않는다. 대기/제안/수행 목록은 `주간현황`의 상세 테이블과 `인력 투입 현황` 상태 필터로 확인한다.

### 1.3 관리자 영역의 인력 관리

```text
관리
└─ 인력 관리
   ├─ 인력 기본정보
   ├─ 재직 상태
   ├─ 역할/직무
   └─ 연간 재직 MM 기준
```

인력 관리 화면은 DB 기준 데이터 역할을 한다.  
운영 화면에서 조회되는 모든 인력 정보는 `관리 > 인력 관리`에 등록된 인력 마스터를 기준으로 한다.

실제 라우트는 기존 구현과의 연속성을 위해 `/people/employment`를 유지한다.  
단, 사이드바에서는 `인력` 메뉴가 아니라 `관리 > 인력 관리`에만 노출한다.

## 2. 시트별 목적/역할/웹 구현 방향

### 2.1 인력재직현황

| 항목 | 내용 |
|---|---|
| 시트 성격 | 인력 마스터 |
| 목적 | PMO 본부 인력의 기본정보, 소속, 직위, 재직 상태, MM 산정 기준을 관리 |
| 관리자가 보는 포인트 | 전체 인력 수, 재직/휴직/전배/퇴직 상태, 팀별 소속, 연간 재직 MM |
| 필요한 데이터 | 성명, 본부, 팀, 직위, 재직여부, 입사일, 시작일, 만료일, 월별 재직 MM, 비고 |
| 웹 구현 방향 | `관리 > 인력 관리` |
| 비고 | 일반 운영 메뉴가 아니라 관리자 마스터 관리 화면으로 구현 |

## 3. 구현 우선순위 판단

### 3.1 결론

P2는 화면을 먼저 만드는 방식보다, 인력 기준정보 API/모델을 최소 단위로 고정한 뒤 화면을 붙이는 순서가 맞다.

이유는 `인력 관리`가 인력 파트의 DB 기준 데이터 역할을 하기 때문이다.

인력 투입 현황, 인력 상세, 주간현황, 월별가동현황은 모두 인력 마스터를 기준으로 조회·집계된다.

주간현황의 대기/제안/수행 현황 테이블도 동일한 인력 마스터와 스냅샷 데이터를 기준으로 한다.

또한 공휴일 관리는 단순 관리자 부가 기능이 아니라 월별 MM, 가동률, 가득률 계산의 기반 데이터다.

따라서 인력 투입 현황과 KPI/보고 화면을 구현하기 전에 공휴일 관리 API/화면을 먼저 정리한다.

### 3.2 권장 구현 순서

```text
1단계. 인력 기준정보 API/모델 정리
   - personnel 조회/등록/수정 API
   - roles 조회/등록/수정 API
   - monthly_employment_mm 조회/수정 API
   - 재직 상태, 역할/직무 기준값 고정

2단계. 관리자 > 인력 관리
   - personnel 목록/등록/수정 UI
   - 역할/직무 기준값 관리 UI
   - 재직 상태 관리 UI
   - 연간 재직 MM 조회/관리 UI

3단계. 공휴일 관리 API/화면
   - holidays 조회/등록/수정/삭제
   - 월별 MM 계산에서 제외할 휴일 기준 관리
   - 월마감/스냅샷 생성 전제 데이터 정리

4단계. 인력 배정 원장 API
   - project_assignments 조회/등록/수정
   - monthly_assignment_mm 조회
   - 중복 배정 경고
   - 기간 검증

5단계. 인력 > 인력 투입 현황
   - 기준일 현재 스냅샷
   - KPI 카드
   - KPI 카드 클릭 시 수행/제안/지원/대기/휴직/전배/퇴직 상태별 인원 목록 필터
   - 현재 인원 목록
   - 차기 투입 예정
   - 인력 상세 진입

6단계. 인력 > 인력 상세
   - 특정 인력의 현재/과거/예정 배정 이력
   - 월별 MM
   - 배정 상세 수정

7단계. KPI/보고 > 주간현황
   - 기준주별 인원 수, 수행/제안/대기, 가동률/가득률
   - current_assignment_snapshots, weekly_load_snapshots 기반 파생 응답

8단계. KPI/보고 > 월별가동현황
   - monthly_employment_mm, monthly_assignment_mm, holidays 기반 월별 MM 집계
   - 월별 가동률/가득률 추이
   - 각 화면 엑셀 다운로드 표준화

대기/제안/수행 상세 목록
   - 별도 페이지가 아니라 KPI/보고 > 주간현황 안에 구현
   - 대기 인원 현황, 제안 인원 현황, 수행 인원 현황 테이블 제공
   - 엑셀 다운로드 시 각 테이블을 별도 시트로 저장
```

### 3.3 MVP 구현 범위

| 구분 | MVP 포함 | MVP 제외 |
|---|---|---|
| 인력 기본정보 | 포함 | 조직 마스터 정규화 |
| 역할/직무 기준 | 포함 | 역할별 단가/원가 |
| 공휴일 관리 | 포함. MM 계산 기반 데이터 | 완전한 월마감 워크플로우 |
| 인력 배정 등록/수정 | 포함 | 드래그앤드롭 배정 |
| 기준일 현재 스냅샷 | 포함. KPI 카드 클릭 상태 필터 포함 | 복잡한 캘린더 편집 |
| 대기/제안 현황 | 인력 투입 현황의 상태 필터와 주간현황 상세 테이블로 대체 | 별도 페이지 구현하지 않음 |
| 월별 MM 집계 | 포함 | 휴가/반차 세부 반영 |
| 중복 배정 경고 | 포함 | 복잡한 과부하 알고리즘 |

### 3.4 인력 관리 완료 후 선행 게이트

`관리 > 인력 관리`는 단순 화면 구현이 아니라 P2 인력 운영의 기준 데이터 확정 단계다.

다음 항목은 인력 관리 페이지와 인력 기준정보 API가 완료된 뒤 진행한다.

1. **인력 마스터 식별자 확정**
   - 모든 인력은 `personnel.id`를 기준 식별자로 사용한다.
   - 성명, 이메일, 직위 같은 표시값은 참조 키로 사용하지 않는다.
   - 동명이인 또는 조직 이동이 있어도 기존 배정/스냅샷/이력이 깨지지 않아야 한다.

2. **PM 포함 프로젝트-인력 참조 정리**
   - 프로젝트의 제안PM, 발표PM, 수행PM, 참여 인력 참조를 가능한 범위에서 `personnel.id` 기반으로 정리한다.
   - 기존 문자열 PM 필드는 표시/마이그레이션 보조값으로만 사용하고, 신규 구현의 기준 참조는 ID로 둔다.
   - 프로젝트 상세, 업무수행현황, 인력 투입 현황에서 같은 인력을 동일 ID로 식별해야 한다.

3. **프로젝트별 인력 배정 기능 구현**
   - 프로젝트별 배정 등록/수정은 인력 마스터 ID와 프로젝트 ID가 확정된 뒤 구현한다.
   - `project_assignments.personnel_id`, `project_assignments.project_id`를 기준으로 배정 원장을 구성한다.

4. **프로젝트 상세 정합성 일괄 마감**
   - 프로젝트 상세의 누적 MM, KPI, 참여 인력 테이블은 인력 배정 모델 전환 후 일괄 정합성을 맞춘다.
   - PM/PL/참여 역할, 배정 기간, 월별 MM, 프로젝트 KPI가 같은 배정 원장을 기준으로 계산되어야 한다.

## 4. 페이지 1 — 관리 > 인력 관리

### 4.1 화면 역할

인력 기본정보, 재직 상태, 역할/직무 기준값을 관리하는 관리자용 마스터 화면이다.

이 화면은 운영 화면이 아니라 DB 기준 데이터 관리 화면이다.  
따라서 정확성, 수정 이력, 필수값 검증이 중요하다.

실제 라우트는 `/people/employment`를 유지한다.  
다만 메뉴 노출 위치는 `관리 > 인력 관리`로 제한하며, 일반 `인력` 메뉴에는 노출하지 않는다.

### 4.2 사용자

| 사용자 | 사용 목적 |
|---|---|
| 관리자 | 인력 등록/수정, 재직 상태 변경, 역할/직무 기준 관리 |
| 본부장 | 전체 인력 현황 조회 |
| 팀장 | 소속 팀 인력 조회 |
| 일반 사용자 | 접근 제한 또는 조회 제한 |

### 4.3 화면 구성

| 영역 | 내용 |
|---|---|
| 상단 필터 | 본부, 팀, 재직여부, 직위, 기준일, 기준연도, 검색어 |
| 액션 버튼 | 조회, 초기화, 엑셀 다운로드, 신규 인력 등록 |
| KPI 카드 | 전체 인력, 재직, 휴직, 전배, 퇴직, 연간 재직 MM |
| 인력 목록 | 인력 마스터 테이블 |
| 인력 상세/수정 패널 | 선택 인력의 기본정보 수정 |
| 역할/직무 관리 탭 | 역할/직무 기준값 관리 |
| 월별 재직 MM 영역 | 기준연도 월별 재직 MM 조회 |

### 4.3.1 주요 기능

- 신규 인력을 등록할 수 있다.
- 기존 인력의 기본정보, 소속, 직위, 재직 상태, MM 산정 기준을 수정할 수 있다.
- 신규 인력 등록 모달/패널은 `프로젝트 관리` 페이지의 등록 UI 흐름, 버튼 배치, 필수값 표시, 저장/취소 동작을 최대한 동일하게 재사용한다.
- 인력 수정 모달/패널은 `프로젝트 관리` 페이지의 수정 UI 흐름과 동일한 형태로 구현한다.
- 인력 목록 테이블은 `프로젝트 관리` 페이지에서 구현된 테이블 요소를 활용하여 헤더 그룹핑과 고정 영역을 적용한다.
- 인력 목록 테이블은 가로 스크롤 시 인력 식별/상태 확인에 필요한 주요 컬럼이 유지되어야 한다.
- 신규 등록, 수정, 테이블 헤더 그룹핑, 고정 컬럼은 새 패턴을 만들지 않고 `프로젝트 관리` 페이지의 구현 요소를 우선 재사용한다.

## 5. 페이지 2 — 인력 > 인력 투입 현황

### 5.1 화면 역할

기준일 현재 PMO 본부 인력이 어떤 상태인지 한눈에 확인하는 인력 운영 메인 화면이다.

이 화면은 기존 `인원별_투입(현재)` 시트의 역할을 웹으로 옮긴다.  
다만 단순 스냅샷만 보여주는 것이 아니라, 인력 상세와 배정 원장으로 진입하는 허브 역할도 한다.

실제 라우트는 `/people/current-assignments`로 구현한다.  
기존 `/people/current` 또는 `/people/assignments` 라우트는 신규 메뉴 라우트로 사용하지 않는다.

### 5.2 사용자

| 사용자 | 사용 목적 |
|---|---|
| 본부장 | 전체 인력의 현재 투입 상태 확인 |
| 팀장 | 팀별 수행/제안/대기 인력 확인 |
| PMO 운영자 | 배정 현황 확인 및 수정 진입 |
| 일반 사용자 | 권한 범위 내 조회 |

### 5.3 화면 구성

| 영역 | 내용 |
|---|---|
| 상단 필터 | 기준일, 본부, 팀, 대표상태, 검색어 |
| 액션 버튼 | 조회, 초기화, 엑셀 다운로드 |
| KPI 카드 | 전체, 수행, 제안, 지원, 대기, 휴직, 전배, 퇴직, 가동률 |
| 현재 인원 스냅샷 | 기준일 현재 인력별 대표 상태 목록 |
| 상태별 인원 요약 | 수행/제안/지원/대기/휴직/전배/퇴직 수량과 비율 |
| 차기 투입 예정 | 기준일 이후 시작 예정인 배정 |
| 월별 MM 요약 | 기준월 또는 선택월 기준 MM 요약 |
| 인력 상세 진입 | 인력명 또는 보기 클릭 시 숨김 라우트 진입 |

### 5.4 필터

| 필터 | 설명 | 기본값 |
|---|---|---|
| 기준일 | 현재 스냅샷 기준일 | 오늘 또는 시스템 기준일 |
| 본부 | 인력 소속 본부 | 전체 |
| 팀 | 인력 소속 팀 | 전체 |
| 대표상태 | 수행/제안/지원/대기/휴직/전배/퇴직 | 전체 |
| 검색어 | 성명, 역할, 프로젝트명 검색 | 없음 |

### 5.5 KPI 카드

| 카드 | 산정 기준 |
|---|---|
| 전체 인원 | 기준일 현재 조회 대상 인력 수 |
| 수행 | 대표 상태가 수행인 인력 수 |
| 제안 | 대표 상태가 제안인 인력 수 |
| 지원 | 대표 상태가 지원인 인력 수 |
| 대기 | 대표 상태가 대기인 인력 수 |
| 휴직 | 재직상태가 휴직인 인력 수 |
| 전배 | 재직상태가 전배인 인력 수 |
| 퇴직 | 재직상태가 퇴직인 인력 수 |
| 가동률 | `(수행 + 제안 + 지원) / 전체 대상 인원 × 100` |

### 5.6 현재 인원 스냅샷 컬럼

| 컬럼 | 설명 |
|---|---|
| No | 행 번호 |
| 성명 | 인력명 |
| 본부/팀 | 소속 |
| 대표상태 | 수행/제안/지원/대기/휴직/전배/퇴직 |
| 제안/수행 프로젝트 | 기준일 현재 제안 또는 수행 대표 프로젝트. 사업명 클릭 시 프로젝트 상세로 이동 |
| 현재 구분 | 수행/제안/지원/대기 |
| 역할 | 제안/수행 프로젝트에서 맡은 역할. 대표상태가 제안/수행이 아니면 `-` |
| 참여기간 | 현재 배정 시작일~종료일 |
| 대기 시작일 | 대기 상태 시작일과 총 대기일. 예: `2026-06-01 (11일)` |
| 상주여부 | 상주/원격/혼합 |
| 차기 예정 | 다음 배정 프로젝트 |
| 시작예정일 | 다음 배정 시작일 |
| 인력 상세 | 화살표 클릭 시 인력 상세로 이동 |

### 5.7 동작 규칙

- 조회 대상은 PMO본부 인력으로 제한한다.
- 기본 정렬은 직급순이며, 같은 직급 안에서는 성명 가나다순으로 정렬한다.
- 기준일 현재 배정이 있는 경우 대표 상태를 배정유형 기준으로 표시한다.
- 배정이 없고 재직 상태가 active이면 대기로 표시한다.
- 휴직/전배/퇴직은 배정 여부보다 우선한다.
- 대표상태가 제안 또는 수행이면 `제안/수행 프로젝트` 컬럼에 해당 사업명을 표시한다.
- 대표상태가 제안 또는 수행이 아니면 `제안/수행 프로젝트`와 `역할`은 `-`로 표시한다.
- `제안/수행 프로젝트` 컬럼의 사업명 클릭 시 해당 프로젝트 상세 페이지로 이동한다.
- `인력 상세` 컬럼의 화살표 클릭 시 해당 인력 상세 페이지로 이동한다.
- 대기 시작일은 기준일 현재 대기 상태가 시작된 날짜를 표시하고, 기준일까지의 총 대기일을 함께 표시한다.
- 한 인력에게 여러 배정이 겹칠 경우 대표 프로젝트 1개를 표시하고 중복 표시 또는 경고 배지를 표시한다.
- 행 클릭 시 `인력 상세`로 이동한다.
- 이 화면에서는 직접 복잡한 배정 수정을 하지 않고 상세 화면 또는 모달로 이동한다.

### 5.8 API

```http
GET /api/current-assignment-snapshots
POST /api/current-assignment-snapshots/calculate
GET /api/personnel
GET /api/project-assignments
GET /api/monthly-assignment-mm
```

`POST /api/current-assignment-snapshots/calculate`는 기준일 스냅샷 계산 API다.

요청 기준일을 받아 PMO본부 인력의 대표 상태, 제안/수행 프로젝트, 역할, 대기 시작일, 총 대기일, 차기 예정일을 계산한다.

계산 결과는 `current_assignment_snapshots`에 저장하거나 즉시 응답할 수 있으나, 같은 기준일/같은 데이터 조건에서는 화면 조회 결과와 동일해야 한다.

### 5.9 DoD

- 기준일 현재 인력별 대표 상태가 조회된다.
- KPI 카드와 목록 수치가 일치한다.
- PMO본부 인력만 조회된다.
- 직급순, 같은 직급 내 성명 가나다순으로 정렬된다.
- 본부/팀/상태/검색어 필터가 동작한다.
- KPI 카드 클릭 시 해당 상태로 목록이 필터링된다.
- 제안/수행 프로젝트 사업명 클릭 시 프로젝트 상세로 이동한다.
- 인력 상세 화살표 클릭 시 인력 상세로 이동한다.
- 대기 인력은 대기 시작일과 총 대기일이 표시된다.
- 차기 투입 예정이 표시된다.
- 엑셀 다운로드가 가능하다.

## 6. 숨김 라우트 — 인력 상세

### 6.1 화면 역할

특정 인력 1명의 투입 이력, 현재 배정, 예정 배정, 월별 MM을 확인하고 배정 정보를 수정하는 상세 화면이다.

이 화면은 기존 `인력_투입현황MM` 시트의 내용을 인력별로 재구성한 상세 화면이다.

### 6.2 라우트

```text
/people/assignments/[personnelId]
```

또는 프로젝트 구조에 맞춰 다음 형태를 사용할 수 있다.

```text
/people/personnel/[personnelId]/assignments
```

사이드바에는 노출하지 않는다.

### 6.3 진입 경로

```text
인력 > 인력 투입 현황
→ 특정 인력 행 클릭
→ 인력 상세
```

### 6.4 화면 구성

| 영역 | 내용 |
|---|---|
| 인력 요약 | 성명, 본부, 팀, 직위, 재직상태, 현재 대표상태 |
| 현재 배정 | 기준일 현재 프로젝트, 역할, 기간, 상주여부 |
| 배정 이력 목록 | 과거/현재/예정 배정 전체 |
| 월별 MM | 기준연도 월별 투입 MM. 연도 선택 필터 제공 |
| 차기 예정 | 다음 배정 프로젝트와 시작일 |
| 배정 등록/수정 | 신규 배정 등록, 기존 배정 수정 |
| 진행 메모 | 인력 배정 관련 메모 또는 프로젝트 진행이력 연결 |

### 6.5 월별 MM 테이블

| 항목 | 내용 |
|---|---|
| 필터 | 기준연도 선택 |
| 컬럼 | 1월~12월, 합계 |
| 데이터 | 월별 수행/제안/지원/미투입 MM |
| 산정 기준 | `monthly_assignment_mm`, `monthly_employment_mm`, `holidays` |

월별 MM 테이블은 기준연도 선택 시 해당 연도의 월별 값을 다시 조회한다.

### 6.6 투입 인력/참여 정보 테이블 컬럼

| 컬럼 | 설명 |
|---|---|
| 프로젝트코드 | 연결 프로젝트 코드 |
| 프로젝트명 | 프로젝트명. 사업명 클릭 시 프로젝트 상세로 이동 |
| 배정유형 | 수행/제안/지원/미투입 |
| 역할 | PM, PL, 개발 등 |
| 시작일 | 배정 시작일 |
| 종료일 | 배정 종료일 |
| 이번달 MM | 기준월 MM |
| 상주여부 | 상주/원격/혼합 |
| 상태 | planned/assigned/ended/cancelled |
| 비고 | 메모 |
| 작업 | 수정 |

### 6.7 등록/수정 필수값

| 항목 | 필수 여부 |
|---|---|
| 인력 | 필수. 상세 화면에서는 자동 지정 |
| 프로젝트 | 필수 |
| 배정유형 | 필수 |
| 역할 | 필수 또는 권장 |
| 시작일 | 필수 |
| 종료일 | 필수 |
| 상주여부 | 선택 |
| 비고 | 선택 |

### 6.8 검증 규칙

- 종료일은 시작일보다 빠를 수 없다.
- 같은 인력에게 동일 기간에 2개 이상 배정이 있으면 중복 배정 경고를 표시한다.
- MVP에서는 중복 배정을 기본 차단하지 않고 경고 후 저장 가능하도록 한다.
- 단, 동일 프로젝트·동일 인력·동일 기간의 완전 중복은 저장 차단을 권장한다.
- 종료된 배정은 수정 시 경고를 표시한다.
- 프로젝트명 클릭 시 해당 프로젝트 상세 페이지로 이동한다.

### 6.9 API

```http
GET    /api/personnel/{id}
GET    /api/project-assignments?personnelId={id}
POST   /api/project-assignments
PATCH  /api/project-assignments/{id}
GET    /api/monthly-assignment-mm?personnelId={id}
```

### 6.10 DoD

- 특정 인력의 배정 이력을 조회할 수 있다.
- 현재 배정과 예정 배정을 구분해 볼 수 있다.
- 기준연도 선택 필터로 월별 MM이 조회된다.
- 참여 정보 테이블의 프로젝트명 클릭 시 프로젝트 상세로 이동한다.
- 신규 배정을 등록할 수 있다.
- 기존 배정을 수정할 수 있다.
- 중복 배정 경고가 표시된다.
- 목록으로 버튼 클릭 시 인력 투입 현황으로 복귀한다.

## 7. KPI/보고 연계

### 7.1 주간현황

주간현황은 KPI 보고 화면이다.  
MVP에서 대기/제안 인원 상세 리스트는 `인력 > 인력 투입 현황`의 상태 KPI 카드 필터 결과로 확인한다.

주간 KPI API는 `GET /api/weekly-kpi-summaries`로 제공한다.  
MVP에서는 별도 `weekly_kpi_summaries` 테이블을 만들지 않고 `current_assignment_snapshots`, `weekly_load_snapshots` 기반 파생 응답으로 구현한다.

주간현황의 역할은 다음으로 제한한다.

| 영역 | 내용 |
|---|---|
| 기준주 필터 | 기준주, 본부, 팀 |
| KPI 카드 | 현재 인원, 계약/수행 인원, 제안 인원, 비가동 인원, 가동률, 가득률 |
| 주간 요약 표 | 본부/팀별 인원 수와 비율 |
| 대기 인원 현황 | 기준주 현재 대기 인원 목록 |
| 제안 인원 현황 | 기준주 현재 제안 참여 인원 목록 |
| 수행 인원 현황 | 기준주 현재 수행 참여 인원 목록 |
| 엑셀 다운로드 | 주간 요약 현황, 대기 인원 현황, 제안 인원 현황, 수행 인원 현황을 각각 별도 시트로 저장 |

대기 인원 현황 테이블은 다음 컬럼을 포함한다.

| 컬럼 | 설명 |
|---|---|
| 성명 | 인력명 |
| 본부/팀 | 소속 |
| 직위/직급 | 직위 또는 직급 |
| 대기 시작일 | 대기 시작일과 총 대기일. 예: `2026-06-10 (2일)` |
| 차기 예정 프로젝트 | 예정 배정이 있을 경우 프로젝트명 |
| 시작예정일 | 차기 배정 시작일 |
| 비고 | 운영 메모 |

주간현황 엑셀 다운로드는 다음 시트를 생성한다.

| 시트명 | 내용 |
|---|---|
| 주간 요약 현황 | 본부/팀별 KPI 요약 |
| 대기 인원 현황 | 기준주 대기 인원 상세 |
| 제안 인원 현황 | 기준주 제안 참여 인원 상세 |
| 수행 인원 현황 | 기준주 수행 참여 인원 상세 |

### 7.2 월별가동현황

월별가동현황은 `월별투입현황` 시트의 결과를 보고 화면으로 제공한다.

요청 용어가 `월간현황`으로 표현되더라도 DTL과 메뉴명은 `월별가동현황`으로 통일한다.

| 영역 | 내용 |
|---|---|
| 기준연도/기준월 필터 | 연도, 월, 본부, 팀 |
| 비교 옵션 | 체크박스: 일반 / Sold 동시 비교 |
| KPI 카드 | 평균 가동률, 평균 가득률, 수행 MM, 제안 MM, 지원 MM, 미투입 MM |
| 추이 차트 | 월별 가동률/가득률 |
| 월별 상세 데이터 테이블 | 엑셀의 월별투입현황 테이블 구조를 참고하여 수행/제안/지원/미투입/합계 MM, 전월 대비를 표시 |
| 데이터 소스 | 백엔드 집계 결과 |

일반 / Sold 동시 비교 체크박스 동작은 다음과 같다.

- 기본값은 일반 단일 표 표시다.
- 체크 시 표를 2단으로 표시한다.
- 위 표는 일반 기준, 아래 표는 Sold 기준으로 표시한다.
- 일반과 Sold의 정확한 산식과 데이터 포함 범위는 KPI 산식 확정 시 최종 고정한다.
- 산식 확정 전까지는 API 응답에서 `normal`, `sold` 구분 필드를 둘 수 있도록 구조를 열어둔다.

## 10. 계산/집계 규칙

### 10.1 기준일 현재 대표 상태 우선순위

한 인력에게 여러 상태가 겹칠 경우 아래 우선순위를 적용한다.

```text
퇴직 > 휴직 > 전배 > 수행 > 제안 > 지원 > 대기
```

단, 전배/퇴직자는 기본적으로 운영 투입 대상에서 제외한다.

### 10.2 대기 인원 산정

```text
대기 인원 =
기준일 현재 재직(active) 인력
- 수행 배정 인력
- 제안 배정 인력
- 지원 배정 인력
```

휴직/전배/퇴직자는 대기 인원에 포함하지 않는다.

### 10.3 기준일 현재 스냅샷 계산

`인원별_투입(현재)` 성격의 기준일 스냅샷은 백엔드 API에서 계산한다.

계산 기준은 다음과 같다.

- 조회 대상은 PMO본부 인력으로 제한한다.
- 기준일 현재 인력별 대표 상태를 계산한다.
- 대표 상태 우선순위는 `10.1 기준일 현재 대표 상태 우선순위`를 따른다.
- 대표상태가 제안 또는 수행이면 대표 프로젝트와 해당 배정 역할을 함께 계산한다.
- 대표상태가 제안 또는 수행이 아니면 역할은 `-`로 계산한다.
- 대기 상태인 경우 대기 시작일과 총 대기일을 계산한다.
- 차기 예정 프로젝트와 시작예정일을 계산한다.
- 결과 정렬은 직급순, 같은 직급 내 성명 가나다순을 기본값으로 한다.

대기 시작일은 기준일 이전부터 배정이 없고 재직 상태가 active인 기간의 시작일로 산정한다.

직전 배정 종료일 다음 영업일, 인력 MM 시작일, 입사일 중 운영 정책에 맞는 기준을 사용하되, 최종 기준은 API 산식에 고정한다.

### 10.4 월별 재직 MM

- 기준월 내 재직 가능 기간을 계산한다.
- 주말과 공휴일은 제외한다.
- 입사일, 시작일, 만료일, 퇴직/전배/휴직 상태를 반영한다.
- 결과는 `monthly_employment_mm`에 저장하거나 조회 시 백엔드에서 산정한다.

### 10.5 월별 투입 MM

- 배정 시작일과 종료일이 기준월과 겹치는 기간을 계산한다.
- 주말과 공휴일은 제외한다.
- 배정유형별로 수행/제안/지원 MM을 구분한다.
- 프론트에서 직접 산정하지 않고 백엔드에서 계산 또는 집계한다.

### 10.6 가동률

MVP 기준은 다음과 같다.

```text
가동률 = (수행 인원 + 제안 인원 + 지원 인원) / 전체 대상 인원 × 100
```

최종 공식은 공통 KPI DTL 및 PRD 공식이 우선한다.

### 10.7 가득률

MVP 기준은 다음과 같다.

```text
가득률 = 투입 MM / 재직 가능 MM × 100
```

월별 보고에서는 `monthly_assignment_mm`, `monthly_employment_mm`, `monthly_kpi_summaries`를 기준으로 산정한다.

### 10.8 월별가동현황 일반/Sold 비교

월별가동현황은 기본적으로 일반 기준 집계를 표시한다.

일반 / Sold 동시 비교 옵션을 선택하면 같은 기간과 같은 필터 조건으로 일반 기준과 Sold 기준을 함께 비교한다.

- 일반 기준: 최종 확정된 기본 KPI 산식
- Sold 기준: Sold 포함 범위와 산식 확정 후 적용

Sold 산식이 확정되기 전까지는 UI와 API 구조만 비교 가능하게 열어두고, 실제 수치 계산 기준은 후속 확정 항목으로 둔다.

## 11. API 설계

### 11.1 관리자 > 인력 관리 API

```http
GET    /api/personnel
POST   /api/personnel
PATCH  /api/personnel/{id}

GET    /api/monthly-employment-mm
GET    /api/roles
POST   /api/roles
PATCH  /api/roles/{id}
```

### 11.2 인력 투입 현황 API

```http
GET /api/current-assignment-snapshots
POST /api/current-assignment-snapshots/calculate
GET /api/project-assignments
GET /api/monthly-assignment-mm
GET /api/project-assignments/summary
```

`POST /api/current-assignment-snapshots/calculate`는 기준일을 입력받아 PMO본부 인력의 기준일 스냅샷을 계산한다.

응답 또는 저장 결과에는 최소한 다음 값이 포함되어야 한다.

- personnel_id
- 대표상태
- 제안/수행 프로젝트 ID와 표시명
- 역할
- 참여기간
- 대기 시작일
- 총 대기일
- 차기 예정 프로젝트 ID와 표시명
- 시작예정일

### 11.3 인력 상세 API

```http
GET    /api/personnel/{id}
GET    /api/project-assignments?personnelId={id}
POST   /api/project-assignments
PATCH  /api/project-assignments/{id}
GET    /api/monthly-assignment-mm?personnelId={id}
```

### 11.4 KPI/보고 연계 API

```http
GET /api/weekly-kpi-summaries
GET /api/weekly-kpi-summaries/export
GET /api/monthly-kpi-summaries
GET /api/monthly-assignment-mm
```

`GET /api/weekly-kpi-summaries`는 MVP에서 별도 저장 테이블을 만들지 않는다.  
응답은 `current_assignment_snapshots`, `weekly_load_snapshots`를 기준으로 백엔드에서 파생 계산한다.

`GET /api/monthly-kpi-summaries`는 `monthly_kpi_summaries` 저장 집계를 기준으로 한다.  
월별 MM 상세는 `monthly_assignment_mm`, `monthly_employment_mm`를 함께 사용한다.

`GET /api/weekly-kpi-summaries/export`는 주간현황 엑셀 다운로드를 위한 API 또는 프론트 다운로드 데이터 소스다.

엑셀 파일은 `주간 요약 현황`, `대기 인원 현황`, `제안 인원 현황`, `수행 인원 현황` 시트를 포함해야 한다.

`GET /api/monthly-kpi-summaries`는 일반/Sold 비교 옵션을 지원할 수 있도록 응답 구조에 `normal`, `sold` 구분을 둘 수 있다.

Sold 산식이 확정되기 전에는 `sold` 값을 비활성 또는 빈 값으로 응답할 수 있다.

### 11.6 응답 envelope

P1에서 확정한 표준을 유지한다.

{
  "data": {},
  "meta": {
    "page": 1,
    "pageSize": 20,
    "total": 0
  },
  "error": null
}

## 16. 수동 검증 시나리오

### 16.1 관리자 > 인력 관리

5. 신규 인력 등록이 가능한지 확인한다.
6. 기존 인력의 재직 상태를 변경할 수 있는지 확인한다.
7. 역할/직무 기준값을 조회/수정할 수 있는지 확인한다.
8. 월별 재직 MM이 조회되는지 확인한다.
9. 인력 목록 테이블의 헤더 그룹핑과 좌측/우측/헤더 고정이 동작하는지 확인한다.

### 16.2 인력 > 인력 투입 현황

1. 인력 > 인력 투입 현황에 접속한다.
2. 기준일을 선택하고 조회한다.
3. KPI 카드와 현재 인원 스냅샷 수치가 일치하는지 확인한다.
4. 수행/제안/지원/대기 KPI 카드를 클릭하면 해당 상태로 인원 목록이 필터링되는지 확인한다.
5. 본부/팀/대표상태/검색어 필터가 동작하는지 확인한다.
6. PMO본부 인력만 표시되는지 확인한다.
7. 직급순, 같은 직급 내 성명 가나다순으로 정렬되는지 확인한다.
8. 제안/수행 상태 인력의 프로젝트 사업명 클릭 시 프로젝트 상세로 이동하는지 확인한다.

10. 대기 인력의 대기 시작일과 총 대기일이 표시되는지 확인한다.
11. 인력 상세 화살표 클릭 시 인력 상세로 이동하는지 확인한다.
12. 기준일 스냅샷 계산 API 결과와 화면 조회 결과가 일치하는지 확인한다.

4. 참여 정보 테이블의 프로젝트명 클릭 시 프로젝트 상세로 이동하는지 확인한다.

6. 종료일이 시작일보다 빠른 경우 저장이 차단되는지 확인한다.
7. 겹치는 기간의 배정 등록 시 중복 배정 경고가 표시되는지 확인한다.

## 18. 최종 판단

P2 인력 운영은 다음 구조로 확정한다.

1. **관리 > 인력 관리**
   - 인력재직현황과 역할과직무를 통합한 마스터 관리 화면
   - 인력 파트의 기준 데이터 역할
   - 인력 마스터 식별자와 상태를 확정하는 선행 게이트
   - PM 포함 프로젝트-인력 참조를 ID 기반으로 정리하기 전에 완료해야 한다.

2. **인력 > 인력 투입 현황**
   - 기준일 현재 인력별 대표 상태를 보는 메인 운영 화면
   - 기존 인원별_투입(현재) 시트를 웹화
   - KPI 카드 클릭으로 수행/제안/지원/대기 등 상태별 인원 목록을 필터링
   - 기준일 스냅샷 계산 API를 통해 PMO본부 인력, 제안/수행 프로젝트, 역할, 대기 시작일, 총 대기일을 계산
   - 특정 인력 클릭 시 숨김 상세 라우트로 이동

3. **숨김 라우트 > 인력 상세**
   - 기존 인력_투입현황MM 시트를 인력별 상세로 재구성
   - 배정 등록/수정, 기준연도별 월별 MM, 투입 이력 확인
   - 프로젝트명 클릭 시 프로젝트 상세로 이동

4. **KPI/보고 > 주간현황**
   - 주간 KPI와 대기/제안/수행 인원 현황을 제공
   - 엑셀 다운로드는 주간 요약 현황, 대기 인원 현황, 제안 인원 현황, 수행 인원 현황을 별도 시트로 제공

5. **KPI/보고 > 월별가동현황**
   - 월별투입현황은 별도 화면이 아니라 백엔드 집계/API로 제공
   - 일반 / Sold 동시 비교는 화면 구조와 API 구조를 열어두되, Sold 산식은 후속 확정한다.
   - 프론트 계산은 하지 않는다.

이 구조가 MVP 기준 페이지 수와 운영 복잡도를 줄이면서도, 상태별 인력 파악은 `인력 투입 현황`에서 처리하고 주간 보고용 대기/제안/수행 목록은 `주간현황`에서 처리하는 방향이다.
