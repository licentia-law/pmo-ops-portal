## 0. 최종 변경 요약

### 0.1 이번 업데이트의 핵심 판단

이번 P2 인력 운영은 기존처럼 여러 엑셀 시트를 각각 별도 화면으로 옮기지 않는다.

엑셀 시트의 성격을 다음 4가지로 구분한다.

1. **마스터 데이터**
   - 관리자 화면에서 등록·수정·관리한다.
   - 예: 인력재직현황, 역할과직무.

2. **운영 원장 데이터**
   - 실제 인력 배정의 기준 데이터다.
   - 별도 사이드바 메뉴보다는 운영 화면 또는 상세 화면의 데이터 소스로 사용한다.
   - 예: 인력_투입현황MM.

3. **운영 스냅샷/현황판**
   - 본부장/팀장/운영자가 자주 보는 화면이다.
   - 메뉴에 노출한다.
   - 예: 인원별_투입(현재), 대기현황, Ⅱ.3.대기·제안인원.

4. **보고/집계 데이터**
   - 별도 입력 화면이 아니라 백엔드 집계/API 또는 KPI/보고 화면의 데이터 소스로 둔다.
   - 예: 월별투입현황, Ⅱ.1.주간현황, Ⅱ.2.월별가동현황.

| 항목 | 기존 1차 업데이트 | 이번 2차 업데이트 |

### 0.4 3차 업데이트 확정 사항

| 항목 | 확정 방향 |
|---|---|
| 현재 메뉴 구조 | DTL 기준으로 수정한다. 인력 메뉴는 `인력 투입 현황`, `대기/제안 현황` 2개만 노출한다. |
| 인력 관리 라우트 | 기존 `/people/employment`를 유지하되 `관리 > 인력 관리` 메뉴에만 노출한다. |
| 인력 투입 현황 라우트 | `/people/current-assignments`로 구현한다. |
| 대기/제안 현황 라우트 | `/people/waiting-proposals`로 구현한다. |
| 인력 화면/API 현황 | 현재 mock 또는 P1성 구현은 참고만 한다. 본 문서는 P2 신규 구현 기준이다. |
| 역할/직무 기준 | `roles` 신규 테이블/API로 구현한다. 문자열/enum 임시 처리는 MVP 기준에서 제외한다. |
| 배정 취소 상태 | `AssignmentStatus.cancelled`를 추가한다. 취소된 배정은 삭제하지 않고 이력으로 보존한다. |
| 근무 위치 구분 | `WorkLocationType` enum을 추가한다. 값은 `onsite`, `remote`, `hybrid`로 한다. |
| 주간 KPI | `GET /api/weekly-kpi-summaries` API는 제공하되, MVP에서는 별도 테이블 없이 `current_assignment_snapshots`, `weekly_load_snapshots` 기반 파생 응답으로 구현한다. |

## 1. 최종 메뉴 구조

### 1.1 전체 메뉴

```text
홈
└─ 홈

프로젝트
├─ 업무수행현황
└─ 진행이력

숨김 라우트
└─ 프로젝트 상세
   ├─ 업무수행현황에서 프로젝트 클릭 시 진입
   └─ 사이드바에는 노출하지 않음

인력
├─ 인력 투입 현황
└─ 대기/제안 현황

숨김 라우트
└─ 인력별 투입 상세
   ├─ 인력 투입 현황에서 인력 클릭 시 진입
   └─ 사이드바에는 노출하지 않음

KPI/보고
├─ 주간현황
└─ 월별가동현황

관리
├─ 사용자/권한 관리
├─ 기준정보 관리
├─ 프로젝트 관리
├─ 인력 관리 (/people/employment)
├─ 공휴일 관리
└─ 월마감/스냅샷
```

### 1.2 인력 영역 메뉴

```text
인력
├─ 인력 투입 현황
└─ 대기/제안 현황
```

인력 영역은 운영자가 실제로 매일 보는 화면만 둔다.

- `인력 투입 현황`: 기준일 현재 누가 어디에 투입되어 있는지 확인하는 메인 운영 화면
- `대기/제안 현황`: 대기 인력과 제안/수행 참여 인력을 집중 확인하는 운영 보고 화면

### 1.3 관리자 영역의 인력 관리

```text
관리
└─ 인력 관리
   ├─ 인력 기본정보
   ├─ 재직 상태
   ├─ 역할/직무
   └─ 연간 재직 MM 기준
```

인력 관리 화면은 DB 기준 데이터 역할을 한다.  
운영 화면에서 조회되는 모든 인력 정보는 `관리 > 인력 관리`에 등록된 인력 마스터를 기준으로 한다.

실제 라우트는 기존 구현과의 연속성을 위해 `/people/employment`를 유지한다.  
단, 사이드바에서는 `인력` 메뉴가 아니라 `관리 > 인력 관리`에만 노출한다.

## 2. 시트별 목적/역할/웹 구현 방향

### 2.1 인력재직현황

| 항목 | 내용 |
|---|---|
| 시트 성격 | 인력 마스터 |
| 목적 | PMO 본부 인력의 기본정보, 소속, 직위, 재직 상태, MM 산정 기준을 관리 |
| 관리자가 보는 포인트 | 전체 인력 수, 재직/휴직/전배/퇴직 상태, 팀별 소속, 연간 재직 MM |
| 필요한 데이터 | 성명, 본부, 팀, 직위, 재직여부, 입사일, 시작일, 만료일, 월별 재직 MM, 비고 |
| 웹 구현 방향 | `관리 > 인력 관리` |
| 비고 | 일반 운영 메뉴가 아니라 관리자 마스터 관리 화면으로 구현 |

### 2.2 역할과직무

| 항목 | 내용 |
|---|---|
| 시트 성격 | 역할/직무 기준정보 |
| 목적 | 인력 배정 시 사용하는 역할/직무 값을 표준화 |
| 관리자가 보는 포인트 | PM, PL, 개발, 분석, 인프라, QA 등 역할 기준의 일관성 |
| 필요한 데이터 | 역할명, 직무그룹, 설명, 사용여부, 정렬순서 |
| 웹 구현 방향 | `관리 > 인력 관리` 안의 역할/직무 관리 영역 |
| 비고 | 별도 메뉴로 분리하지 않고 인력 관리 안에 통합 |

### 2.3 인력_투입현황MM

| 항목 | 내용 |
|---|---|
| 시트 성격 | 인력 배정 원장 |
| 목적 | 사람 × 프로젝트 × 기간 단위의 투입/제안/지원 배정 정보를 관리 |
| 관리자가 보는 포인트 | 누가 어떤 프로젝트에 언제부터 언제까지 어떤 역할로 배정되었는지 |
| 필요한 데이터 | 인력, 프로젝트, 배정유형, 역할, 시작일, 종료일, 상주여부, 이번달 MM, 월별 MM, 비고 |
| 웹 구현 방향 | `인력 > 인력별 투입 상세`의 핵심 데이터 |
| 비고 | 사이드바에 직접 노출하지 않고 `인력 투입 현황`에서 특정 인력 클릭 시 진입 |

### 2.4 인원별_투입(현재)

| 항목 | 내용 |
|---|---|
| 시트 성격 | 기준일 현재 스냅샷 |
| 목적 | 기준일 현재 인력별 대표 상태와 현재 투입 프로젝트를 확인 |
| 관리자가 보는 포인트 | 현재 수행/제안/지원/대기/휴직/전배/퇴직 인원, 차기 투입 예정 |
| 필요한 데이터 | 기준일, 성명, 본부, 팀, 대표상태, 현재 프로젝트, 역할, 참여기간, 차기 예정 |
| 웹 구현 방향 | `인력 > 인력 투입 현황` |
| 비고 | 인력 운영의 메인 화면으로 사용 |

### 2.5 대기현황

| 항목 | 내용 |
|---|---|
| 시트 성격 | 주간 인력 운영판 |
| 목적 | 전주/금주/차주 기준으로 대기, 투입 예정, 휴직, 전배, 퇴직 상태를 확인 |
| 관리자가 보는 포인트 | 이번 주 대기 인원, 다음 주 투입 예정 인원, 빈 슬롯, 차주 리스크 |

| 웹 구현 방향 | `인력 > 대기/제안 현황` |
| 비고 | 본부장/팀장 확인 빈도가 높으므로 별도 페이지 유지 |

### 2.6 Ⅱ.3.대기·제안인원

| 항목 | 내용 |
|---|---|
| 시트 성격 | 대기/제안 보고 리스트 |

| 필요한 데이터 | 기준일, 대기 인력 목록, 투입 예정 사업, 제안/수행 프로젝트, 성명, 직위, 투입종료, 인원수 |

### 2.7 월별투입현황

| 비고 | 프론트 스크립트 계산은 금지. `KPI/보고 > 월별가동현황`에서 결과를 조회 |

### 2.8 Ⅱ.1.주간현황

| 항목 | 내용 |
|---|---|
| 시트 성격 | 주간 KPI 보고 |
| 목적 | 기준주 기준 본부/팀별 인원 수와 가동률/가득률을 요약 |
| 관리자가 보는 포인트 | 현재 인원, 수행 인원, 제안 인원, 비가동 인원, 가동률, 가득률 |
| 필요한 데이터 | 기준주, 본부, 팀, 현재 인원, 수행, 제안, 대기, 비가동, 가동률, 가득률 |
| 웹 구현 방향 | `KPI/보고 > 주간현황` |
| 비고 | 대기/제안 상세는 `인력 > 대기/제안 현황`에서 확인하고, 주간현황은 KPI 중심으로 유지 |

### 2.9 Ⅱ.2.월별가동현황

| 항목 | 내용 |
|---|---|
| 시트 성격 | 월별 KPI 보고 |
| 목적 | 월별 MM, 가동률, 가득률 추이를 확인 |

## 3. 구현 우선순위 판단

### 3.1 결론

`관리 > 인력 관리`를 먼저 구현하는 것이 맞다.

이유는 `인력 관리`가 인력 파트의 DB 기준 데이터 역할을 하기 때문이다.  
인력 투입 현황, 인력별 투입 상세, 대기/제안 현황, 주간현황, 월별가동현황은 모두 인력 마스터를 기준으로 조회·집계된다.

### 3.2 권장 구현 순서

```text
1단계. 관리자 > 인력 관리
   - personnel 조회/등록/수정
   - 역할/직무 기준값 조회/수정
   - 재직 상태 관리
   - 연간 재직 MM 조회

2단계. 인력 배정 원장 API
   - project_assignments 조회/등록/수정
   - monthly_assignment_mm 조회
   - 중복 배정 경고
   - 기간 검증

3단계. 인력 > 인력 투입 현황
   - 기준일 현재 스냅샷
   - KPI 카드
   - 현재 인원 목록
   - 차기 투입 예정
   - 인력별 투입 상세 진입

4단계. 인력 > 인력별 투입 상세
   - 특정 인력의 현재/과거/예정 배정 이력
   - 월별 MM
   - 배정 상세 수정

5단계. 인력 > 대기/제안 현황
   - 대기현황 + Ⅱ.3 대기·제안인원 통합
   - 전주/금주/차주 운영판
   - 대기 인원 리스트
   - 제안/수행 인원 리스트

6단계. KPI/보고 연계
   - 주간현황
   - 월별가동현황
   - 각 화면 엑셀 다운로드 표준화
```

### 3.3 MVP 구현 범위

| 구분 | MVP 포함 | MVP 제외 |
|---|---|---|
| 인력 기본정보 | 포함 | 조직 마스터 정규화 |
| 역할/직무 기준 | 포함 | 역할별 단가/원가 |
| 인력 배정 등록/수정 | 포함 | 드래그앤드롭 배정 |
| 기준일 현재 스냅샷 | 포함 | 복잡한 캘린더 편집 |
| 대기/제안 현황 | 포함 | 자동 최적 배정 추천 |
| 월별 MM 집계 | 포함 | 휴가/반차 세부 반영 |
| 중복 배정 경고 | 포함 | 복잡한 과부하 알고리즘 |

## 5. 페이지 2 — 인력 > 인력 투입 현황

다만 단순 스냅샷만 보여주는 것이 아니라, 인력 상세와 배정 원장으로 진입하는 허브 역할도 한다.

실제 라우트는 `/people/current-assignments`로 구현한다.  
기존 `/people/current` 또는 `/people/assignments` 라우트는 신규 메뉴 라우트로 사용하지 않는다.

### 5.2 사용자

## 7. 페이지 3 — 인력 > 대기/제안 현황

### 7.1 화면 역할

대기 인력과 제안/수행 참여 인력을 집중 확인하는 본부장/팀장용 운영 현황 화면이다.

이 화면은 기존 `대기현황`과 `Ⅱ.3.대기·제안인원` 시트를 통합한다.  
주간현황 하단에 흡수하지 않고 별도 페이지로 유지한다.

실제 라우트는 `/people/waiting-proposals`로 구현한다.  
기존 `/people/waiting` 라우트는 신규 메뉴 라우트로 재사용하지 않는다.

### 7.2 별도 페이지로 유지하는 이유

- 본부장/팀장이 대기 인력을 확인하는 빈도가 높다.
- 대기 인력은 단순 KPI가 아니라 운영 의사결정 대상이다.
- 차주 투입 예정, 휴직/전배/퇴직, 제안 참여 현황을 함께 봐야 한다.
- 주간현황에 넣으면 보고 화면이 과도하게 복잡해질 수 있다.

## 10. 계산/집계 규칙

### 10.1 기준일 현재 대표 상태 우선순위

한 인력에게 여러 상태가 겹칠 경우 아래 우선순위를 적용한다.

```text
퇴직 > 휴직 > 전배 > 수행 > 제안 > 지원 > 대기
```

단, 전배/퇴직자는 기본적으로 운영 투입 대상에서 제외한다.

### 10.2 대기 인원 산정

```text
대기 인원 =
기준일 현재 재직(active) 인력
- 수행 배정 인력
- 제안 배정 인력
- 지원 배정 인력
```

휴직/전배/퇴직자는 대기 인원에 포함하지 않는다.

### 10.3 월별 재직 MM

- 기준월 내 재직 가능 기간을 계산한다.
- 주말과 공휴일은 제외한다.
- 입사일, 시작일, 만료일, 퇴직/전배/휴직 상태를 반영한다.
- 결과는 `monthly_employment_mm`에 저장하거나 조회 시 백엔드에서 산정한다.

### 10.4 월별 투입 MM

- 배정 시작일과 종료일이 기준월과 겹치는 기간을 계산한다.
- 주말과 공휴일은 제외한다.
- 배정유형별로 수행/제안/지원 MM을 구분한다.
- 프론트에서 직접 산정하지 않고 백엔드에서 계산 또는 집계한다.

### 10.5 가동률

MVP 기준은 다음과 같다.

```text
가동률 = (수행 인원 + 제안 인원 + 지원 인원) / 전체 대상 인원 × 100
```

최종 공식은 공통 KPI DTL 및 PRD 공식이 우선한다.

### 10.6 가득률

MVP 기준은 다음과 같다.

```text
가득률 = 투입 MM / 재직 가능 MM × 100
```

월별 보고에서는 `monthly_assignment_mm`, `monthly_employment_mm`, `monthly_kpi_summaries`를 기준으로 산정한다.

## 16. 수동 검증 시나리오

### 16.1 관리자 > 인력 관리

1. 관리자 계정으로 로그인한다.
2. 관리 > 인력 관리에 접속한다.
3. 인력 목록이 조회되는지 확인한다.
4. 본부/팀/재직여부/검색어 필터가 동작하는지 확인한다.
5. 신규 인력 등록이 가능한지 확인한다.
6. 기존 인력의 재직 상태를 변경할 수 있는지 확인한다.
7. 역할/직무 기준값을 조회/수정할 수 있는지 확인한다.
8. 월별 재직 MM이 조회되는지 확인한다.

### 16.2 인력 > 인력 투입 현황

1. 인력 > 인력 투입 현황에 접속한다.
2. 기준일을 선택하고 조회한다.
3. KPI 카드와 현재 인원 스냅샷 수치가 일치하는지 확인한다.
4. 본부/팀/대표상태/검색어 필터가 동작하는지 확인한다.
5. 특정 인력 행을 클릭한다.
6. 인력별 투입 상세로 이동하는지 확인한다.

### 16.3 인력별 투입 상세

1. 특정 인력의 배정 이력이 표시되는지 확인한다.
2. 현재 배정, 과거 배정, 예정 배정이 구분되는지 확인한다.
3. 월별 MM이 조회되는지 확인한다.
4. 신규 배정을 등록한다.
5. 종료일이 시작일보다 빠른 경우 저장이 차단되는지 확인한다.
6. 겹치는 기간의 배정 등록 시 중복 배정 경고가 표시되는지 확인한다.
7. 목록으로 버튼 클릭 시 인력 투입 현황으로 복귀하는지 확인한다.

### 16.4 인력 > 대기/제안 현황

1. 인력 > 대기/제안 현황에 접속한다.
2. 기준주를 선택하고 조회한다.
3. 전주/금주/차주 운영판이 표시되는지 확인한다.
4. 대기 인원 리스트가 표시되는지 확인한다.
5. 제안/수행 인원 리스트가 표시되는지 확인한다.
6. 차주 투입 예정 목록이 표시되는지 확인한다.
7. 본부/팀/상태/검색어 필터가 동작하는지 확인한다.
8. 엑셀 다운로드가 가능한지 확인한다.

### 16.5 KPI/보고 연계

1. KPI/보고 > 주간현황에 접속한다.
2. 기준주 기준 KPI가 조회되는지 확인한다.
3. KPI/보고 > 월별가동현황에 접속한다.
4. 월별 MM, 가동률, 가득률이 백엔드 집계 기준으로 표시되는지 확인한다.
5. 주간현황과 월별가동현황 화면의 조회 조건이 엑셀 다운로드에 동일하게 적용되는지 확인한다.

## 18. 최종 판단

P2 인력 운영은 다음 구조로 확정한다.

1. **관리 > 인력 관리**
   - 인력재직현황과 역할과직무를 통합한 마스터 관리 화면
   - 인력 파트의 기준 데이터 역할
   - 가장 먼저 구현 권장

2. **인력 > 인력 투입 현황**
   - 기준일 현재 인력별 대표 상태를 보는 메인 운영 화면
   - 기존 인원별_투입(현재) 시트를 웹화
   - 특정 인력 클릭 시 숨김 상세 라우트로 이동

3. **숨김 라우트 > 인력별 투입 상세**
   - 기존 인력_투입현황MM 시트를 인력별 상세로 재구성
   - 배정 등록/수정, 월별 MM, 투입 이력 확인

4. **인력 > 대기/제안 현황**
   - 대기현황과 Ⅱ.3.대기·제안인원을 통합한 별도 운영 화면
   - 본부장/팀장의 대기 인력 집중 확인 목적

5. **KPI/보고 > 월별가동현황**
   - 월별투입현황은 별도 화면이 아니라 백엔드 집계/API로 제공
   - 프론트 계산은 하지 않는다.

이 구조가 기존 엑셀의 기능을 유지하면서도 MVP 기준 페이지 수와 운영 복잡도를 가장 적절하게 줄이는 방향이다.
