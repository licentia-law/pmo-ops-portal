## 0. 이번 업데이트의 핵심 변경사항

### 0.1 기존 범위
기존 P2 문서는 아래 3개 페이지를 각각 독립 화면으로 구현하는 방향이었다.

1. 인력재직현황
2. 인력배치/투입현황
3. 인원별 투입(현재)

### 0.2 변경 후 범위
대화에서 확정한 페이지 축소 방향에 따라 P2 인력 운영의 MVP 범위는 아래처럼 조정한다.

| 기존 화면/시트 | 변경 후 위치 | 처리 방향 |
|---|---|---|
| 인력재직현황 | 관리자 > 인력 마스터 | 운영 화면이 아니라 기준정보/마스터 관리 성격으로 이동 |
| 역할과직무 | 관리자 > 인력 마스터 | 인력 마스터 안의 역할/직무 기준정보 영역으로 통합 |
| 인력배치/투입현황 | 인력 > 인력배치/투입현황 | P2의 핵심 구현 화면으로 유지 |
| 인원별 투입(현재) | 인력배치/투입현황 내부 | 별도 메뉴 제거, 기준일 현재 스냅샷/요약 영역으로 흡수 |
| 대기현황 | KPI/보고 > 주간현황 | 별도 인력 메뉴로 두지 않고 주간 보고 하단 섹션으로 통합 |
| Ⅱ.3.대기,제안인원 | KPI/보고 > 주간현황 | 대기/제안 인원 현황 테이블로 주간현황에 통합 |

### 0.3 최종 메뉴 기준
P2 완료 후 인력 관련 메뉴는 아래 구조를 기준으로 한다.

```text
인력
└─ 인력배치/투입현황

관리자
└─ 인력 마스터
   ├─ 인력 기본정보
   ├─ 재직 상태
   ├─ 역할/직무
   └─ 연간 재직 MM 기준
```

## 1. P2 목표

### 1.1 비즈니스 목표
인력×프로젝트×기간 단위의 배정 데이터를 관리하고, 기준월/기준일 현재 인력 투입 현황을 조회할 수 있게 한다.

### 1.2 구현 목표
- 인력 배정 데이터를 실제 API/DB 기준으로 조회한다.
- 프로젝트별/인력별 배정 기간, 역할, 구분, 상주여부, 월별 MM을 관리한다.
- 현재 기준일 스냅샷을 통해 누가 어떤 프로젝트에 투입 중인지 확인한다.
- 중복 배정, 미투입, 차기 투입 예정 등 운영상 필요한 상태를 최소 수준으로 표시한다.
- 인력재직현황은 별도 운영 메뉴가 아니라 관리자 영역의 `인력 마스터`로 이동한다.

### 1.3 MVP에서 하지 않는 것
- 사람별 세부 이력 관리 전용 페이지는 만들지 않는다.
- 대기현황 전용 페이지는 만들지 않는다.
- 대기/제안인원 전용 페이지는 만들지 않는다.
- 과도한 워크로드 차트, 드래그 앤 드롭 배정, 캘린더형 편집은 MVP에서 제외한다.
- 인력 상세 프로필 페이지는 MVP에서 필수 구현하지 않는다.

## 2. 페이지 구조

## 2.1 인력배치/투입현황

### 2.1.1 화면 역할
사람과 프로젝트의 배정 관계, 투입 기간, 역할, 구분, 월별 MM을 조회/등록/수정하는 인력 운영의 핵심 화면이다.

### 2.1.2 사용 흐름
```text
인력배치/투입현황 진입
→ 기준월/본부/팀/구분/투입여부 필터
→ 인력별 현재 투입 현황 확인
→ 특정 인력 또는 프로젝트 배정 행 확인
→ 필요 시 신규 배정 등록 또는 기존 배정 수정
→ 월별 MM 및 상태 요약 확인
```

### 2.1.3 화면 구성
상단에서 기준월과 필터를 선택하고, 중앙에는 현재 스냅샷 테이블을 둔다. 하단에는 배정 상세 목록과 월별 MM 요약을 배치한다.

| 영역 | 구성 |
|---|---|
| 상단 필터 | 기준월, 기준일, 본부, 팀, 구분, 투입여부, 상주여부, 검색어 |
| 액션 | 조회, 초기화, 엑셀 다운로드, 신규 배정 등록 |
| KPI 카드 | 전체 인원, 수행, 제안, 지원, 대기, 휴직, 전배, 퇴직, 가동률 |
| 현재 인원 스냅샷 | 기준일 현재 인력별 대표 상태/현재 프로젝트/차기 예정 |
| 배정 상세 목록 | 인력×프로젝트×기간 fact 목록 |
| 월별 MM 요약 | 기준월 또는 연간 월별 MM 요약 |
| 상태별 인원 요약 | 수행/제안/지원/대기/휴직/전배/퇴직 수량 및 비율 |
| 차기 투입 예정 | 다음 시작예정일 기준 배정 예정 인력 |

### 2.1.4 필터 정의

| 필터 | 설명 | 비고 |
|---|---|---|
| 기준월 | 월별 MM 산정 기준월 | 기본값: 현재 월 |
| 기준일 | 현재 스냅샷 산정 기준일 | 기본값: 오늘 또는 시스템 기준일 |
| 본부 | personnel.group_name 기준 | 문자열 조직 기준 |
| 팀 | personnel.team_name 기준 | 문자열 팀 기준 |
| 구분 | 수행/제안/지원/미투입 | DB 내부값 사용 |
| 투입여부 | 투입/미투입 | 배정 행 기준 |
| 상주여부 | 상주/비상주/원격 등 | 프로젝트 배정 기준 |
| 검색어 | 성명, 프로젝트명, 프로젝트코드 검색 | 부분일치 |

### 2.1.5 KPI 카드 정의

| 카드 | 산정 기준 |
|---|---|
| 전체 인원 | 기준일 기준 대상 인력 수 |
| 수행 | 대표 상태 또는 배정유형이 수행인 인력 수 |
| 제안 | 대표 상태 또는 배정유형이 제안인 인력 수 |
| 지원 | 대표 상태 또는 배정유형이 지원인 인력 수 |
| 대기 | 기준일 현재 투입 배정이 없고 대기 상태인 인력 수 |
| 휴직 | 인력 마스터 재직상태가 휴직인 인력 수 |
| 전배 | 인력 마스터 재직상태가 전배인 인력 수 |
| 퇴직 | 인력 마스터 재직상태가 퇴직인 인력 수 |
| 가동률 | `(수행 + 제안 + 지원) / 전체 인원` 기준. 세부 공식은 공통 KPI DTL과 일치시킨다. |

### 2.1.8 동작 규칙
- 신규 배정 등록 시 인력, 프로젝트, 배정유형, 역할, 시작일, 종료일은 필수다.
- 종료일이 시작일보다 빠르면 저장할 수 없다.
- 같은 인력이 같은 기간에 2개 이상 배정되면 중복 배정 경고를 표시한다.
- 중복 배정은 MVP에서는 차단이 아니라 경고로 처리한다.
- 외부 인원 여부가 필요한 경우 비고 또는 별도 플래그로 표시하되, MVP에서는 필수값으로 강제하지 않는다.
- 월별 MM은 시작일/종료일과 공휴일 기준을 사용해 산정한다.
- 월별 MM 계산 공식은 공통 KPI/집계 DTL과 동일해야 한다.
- 타임라인은 조회 보조 UI이며, 실제 수정 진입점은 배정 상세 목록이다.

## 3. 관리자 > 인력 마스터

## 3.1 화면 역할

### 3.1.6 DoD

## 4. 주간현황과의 관계

대기현황과 대기/제안인원은 P2 인력 화면에서 제외하고, KPI/보고의 `주간현황` 하단으로 통합한다.

### 4.1 주간현황 하단 통합 섹션
주간현황 하단에는 아래 2개 테이블을 둔다.

1. 대기 인원 현황
2. 제안 인원 현황

### 4.2 P2에서 제공해야 하는 데이터
P2 인력배치/투입현황은 주간현황이 아래 데이터를 조회할 수 있도록 API 또는 집계 뷰를 제공해야 한다.

| 데이터 | 설명 |
|---|---|
| 기준일 현재 대기 인원 | 기준일 기준 투입 배정이 없고 대기 상태인 인력 |
| 차주 투입 예정 | 기준일 이후 다음 주에 시작하는 배정 |
| 제안 참여 인원 | 배정유형이 proposal인 인력 |
| 제안 프로젝트 수 | proposal 상태 프로젝트 또는 proposal assignment 기준 |
| PT/입찰 예정 | 프로젝트 일정 또는 진행 상태 기반 |

### 4.3 권장 API
```http
GET /api/weekly-load-snapshots
GET /api/current-assignment-snapshots
GET /api/project-assignments?assignmentType=proposal
```

|---|---|
| personnel | 인력 마스터 |
| project_assignments | 인력×프로젝트 배정 fact |
| monthly_employment_mm | 월별 재직 MM |
| monthly_assignment_mm | 월별 투입 MM |
| current_assignment_snapshots | 기준일 현재 스냅샷 |
| weekly_load_snapshots | 주간현황/대기/제안 보고용 스냅샷 |
| project_logs | 배정 변경 또는 프로젝트 진행 기록과 연결 가능 |

### 5.2 주요 enum
| 구분 | 값 |
|---|---|
| EmploymentStatus | active, leave, transferred, retired, waiting |
| AssignmentType | delivery, proposal, support, unassigned |
| AssignmentStatus | planned, assigned, ended, cancelled |
| WorkLocationType | onsite, remote, hybrid |
| OrganizationRole | head, team_lead, member, pm, pl, other |

### 5.3 관계
- `project_assignments.project_id`는 `projects.id`를 참조한다.
- `project_assignments.personnel_id`는 `personnel.id`를 참조한다.
- `monthly_assignment_mm.assignment_id`는 `project_assignments.id`를 참조한다.
- `current_assignment_snapshots.personnel_id`는 `personnel.id`를 참조한다.
- `current_assignment_snapshots.project_id`는 현재 대표 프로젝트가 있을 경우 `projects.id`를 참조한다.

## 6. 계산/집계 규칙

### 6.1 기준일 현재 대표 상태 우선순위
기준일 현재 한 인력에게 여러 상태가 겹칠 경우 아래 우선순위를 적용한다.

```text
퇴직 > 휴직 > 전배 > 수행 > 제안 > 지원 > 대기
```

단, 실제 운영에서 전배/퇴직자는 인력 배정 대상에서 제외하는 것이 원칙이다.

### 6.2 월별 MM
- 기준월 내 배정 시작일~종료일이 겹치는 일수를 계산한다.
- 주말과 공휴일은 제외한다.
- `공휴일 관리`에 등록된 공휴일을 반영한다.
- 월별 MM 산식은 공통 KPI/월별가동현황 DTL과 동일하게 유지한다.

### 6.3 가동률
MVP 기준 가동률은 다음과 같이 사용한다.

```text
가동률 = (수행 인원 + 제안 인원 + 지원 인원) / 전체 대상 인원 × 100
```

단, 최종 산식은 PRD 및 KPI/보고 DTL의 공식이 우선한다.

### 6.4 가득률
MVP 기준 가득률은 다음과 같이 사용한다.

```text
가득률 = 투입 MM / 재직 가능 MM × 100
```

## 12. 수동 검증 시나리오

2. 특정 인력 행 선택

## 14. 최종 판단

P2 인력 운영은 기존 3개 독립 페이지를 모두 구현하지 않고, 다음 2개 축으로 정리한다.

1. **인력배치/투입현황**: 실제 운영자가 매일 보는 인력 배정/투입 현황 화면
2. **관리자 > 인력 마스터**: 인력 기본정보와 역할/직무 기준을 관리하는 관리자 화면

대기/제안 인원 보고는 `주간현황` 하단에 통합하여, MVP 기준 페이지 수와 사용 복잡도를 줄인다.
