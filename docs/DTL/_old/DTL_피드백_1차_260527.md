## 1. 코드 레벨 확인 결과

### 1.1 홈
- API: `apps/api/app/api/routes/p1_screens.py`
  - `/api/p1-screens/home`는 `kpis`, `trend`, `teamHeadcount`, `teamUtilization`, `monthSummary`를 반환한다.
  - `_dashboard_kpis()`는 현재 인원, 수행 인원, 제안 인원, 대기 인원, 가동률, 가득률만 반환한다.
  - `WIN/LOSS` 올해 누적 KPI는 아직 없다.
  - `_team_headcount()`와 `_team_utilization()`은 팀별 rows만 반환한다.
- UI: `apps/web/design/pages/HomePage.tsx`
  - `KPIRow`는 API `kpis`와 `monthSummary` 기반 추가 KPI를 병합해 표시한다.
  - `TrendChart`는 현재 `var(--brand)`와 `var(--info)` 색상으로 가동률/가득률을 표시한다.
  - `TeamHeadcount` 제목은 `팀별 인력 현황`이다.
  - `TeamUtilization` 제목은 `팀별 가동률 / 가득률`이다.
  - 본부 행은 프론트에서 아직 계산/표시하지 않는다.

### 1.2 업무수행현황
- UI: `apps/web/design/pages/ExecutionPage.tsx`
  - `ExecutionFilterState.periodPreset`는 `"all" | "recent3m" | "thisMonth" | "lastMonth" | "thisYear"`만 지원한다.
  - `PeriodPicker`는 select 기반 프리셋만 제공하고 직접 날짜 선택 UI는 없다.
  - 기간 필터 기준은 현재 `row.submission?.datetime`의 날짜 부분이다.
  - 필터 결과는 `filteredRows`로 계산되고, 페이지 표시 데이터는 `pagedRows`다.
  - `getP1Screen("code")`가 페이지 진입 시 즉시 호출되어 모달용 데이터까지 선로드한다.
  - CSV/엑셀 다운로드 버튼과 로직은 없다.

### 1.3 진행이력
- UI: `apps/web/design/pages/HistoryPage.tsx`
  - `PeriodPicker`는 커스텀 드롭다운 프리셋만 제공하고 직접 날짜 선택 UI는 없다.
  - 기간 필터 상태는 이미 `{ periodPreset, from, to }` 구조다.
  - 필터 적용 결과는 `filteredRows`로 계산된다.
  - 기간 기준은 `row.datetime`이다.

### 1.4 프로젝트 관리
- UI: `apps/web/design/pages/CodePage.tsx`
  - `CodeFilterState`는 `status`, `customer`, `owner`, `leadPm`, `use`, `query`만 가진다.
  - 기간 필터 상태와 UI는 없다.
  - 현재 필터 결과는 `filteredRows`이고 코드 내림차순 정렬을 적용한다.
  - CSV/엑셀 다운로드 버튼과 로직은 없다.
  - 내부 등록/편집 모달의 `제출 형식` 옵션은 현재 `온라인`, `오프라인`만 있다.
- 공통 모달: `apps/web/design/components/ProjectMasterEditModal.tsx`
  - `제출 형식` 옵션은 현재 `온라인`, `오프라인`만 있다.
  - 저장 payload는 기존 `submission_format` 문자열 필드를 사용한다.

## 2. 확정 요구사항

### 2.1 홈
- 핵심 현황에 올해 누적 `WIN`, `LOSS` 카드 추가.
- 내부 의미:
  - `WIN` = 수주
  - `LOSS` = 실주
- 브라우저 표시명은 `수주/실주`가 아니라 `WIN/LOSS`로 한다.
- `WIN/LOSS` 집계 기준:
  - 우선 `ProjectLog.next_status` + `ProjectLog.logged_at` 기준 올해 누적.
  - 상태 변경 로그가 없는 데이터는 `Project.status`가 `WIN` 또는 `LOSS`이고 `recent_activity_at` 또는 `updated_at`이 올해인 프로젝트를 fallback으로 포함한다.
- 가동률/가득률 색상은 범례, 라인, 점, 마지막 값 라벨, 팀별 막대까지 서로 명확히 구분되도록 조정한다.
- `팀별 인력 현황` 제목은 `인력 현황`으로 변경한다.
- `팀별 가동률 / 가득률` 제목은 `가동률 / 가득률`로 변경한다.
- 본부 현황은 API row를 추가하지 않고 프론트에서 팀 rows 합산으로 계산한다.

### 2.2 기간 필터
- 업무수행현황, 진행이력, 프로젝트 관리 모두 기존 프리셋을 유지한다.
- 추가 프리셋: `직접 선택`.
- 직접 선택 시 `시작일`, `종료일` date input을 표시한다.
- 시작일만 입력하면 시작일 이후 전체, 종료일만 입력하면 종료일 이전 전체, 둘 다 입력하면 포함 범위로 필터링한다.
- 기준 날짜:
  - 업무수행현황: `row.submission.datetime`
  - 진행이력: `row.datetime`
  - 프로젝트 관리: `row.fromDate` 사업 시작일

### 2.3 다운로드
- 업무수행현황과 프로젝트 관리에 엑셀 다운로드 버튼을 추가한다.
- 버튼 위치: 테이블 우측 상단 필터 정보 우측.
- 다운로드 대상: 현재 적용된 `filteredRows` 전체. 현재 페이지의 `pagedRows`/`visibleRows`가 아니다.
- 파일명:
  - `업무수행현황_YYMMDD_hhmm.xlsx`
  - `프로젝트관리_YYMMDD_hhmm.xlsx`
- 다운로드 포맷은 `.xlsx`이며 공통 모듈(`projectWorkbookExport.ts`)을 사용한다.

### 2.4 프로젝트 등록
- `제출 형식` 옵션에 `이메일`을 추가한다.
- 변경 대상:
  - `ProjectMasterEditModal.tsx`
  - `CodePage.tsx` 내부 모달
- DB/API 스키마 변경은 하지 않는다.

### 2.5 성능
- 업무수행현황의 `getP1Screen("code")` 즉시 호출을 제거하고, 신규/편집 모달이 열릴 때 lazy load한다.
- 빈 화면 `return null` 구간은 기존 디자인에 맞는 lightweight loading 상태로 교체한다.
- API N+1은 실제 병목 확인 후 별도 phase에서 bulk query로 개선한다.

## 3. 구현 Phase

각 phase는 구현 후 개발자가 브라우저에서 확인하고, 확인 완료 후 다음 phase로 진행한다.

### Phase 0. 기준 동작 확인
- 홈, 업무수행현황, 진행이력, 프로젝트 관리 페이지가 현재 상태로 정상 진입되는지 확인한다.
- 업무수행현황과 프로젝트 관리에서 현재 필터/페이지네비게이션/등록 모달 동작을 기록한다.
- 브라우저 확인:
  - 각 페이지 첫 진입 화면이 깨지지 않는다.
  - 콘솔 런타임 오류가 없다.

### Phase 1. 홈 WIN/LOSS API 추가
- `p1_screens.py`에 올해 누적 `WIN/LOSS` 집계 helper를 추가한다.
- `_dashboard_kpis()`에 `yearWin`, `yearLoss` KPI를 추가한다.
- label은 브라우저 표시 기준으로 `WIN`, `LOSS`를 사용한다.
- 브라우저 확인:
  - 홈 핵심 현황에 `WIN`, `LOSS` 카드가 보인다.
  - 숫자와 단위가 중복되지 않는다.

### Phase 2. 홈 차트 색상 대비 개선
- `HomePage.tsx`의 `TrendChart`, `DualBar`, 하단 설명 범례에서 가득률 색상을 `var(--info)`와 명확히 다른 청록/녹색 계열로 변경한다.
- 같은 지표는 범례, 라인, 점, 라벨, 막대 색상을 일관되게 적용한다.
- 브라우저 확인:
  - 가동률과 가득률이 한눈에 구분된다.
  - 차트 라벨이 기존 스타일을 유지한다.

### Phase 3. 홈 인력 현황 본부 행 추가
- `TeamHeadcount` 제목을 `인력 현황`으로 변경한다.
- 기존 `rows`를 합산해 `PMO본부` 행을 상단에 추가한다.
- 본부 행은 배경색, 상하 구분선, 굵은 텍스트 등으로 팀 행과 구분한다.
- 브라우저 확인:
  - `PMO본부` 합계가 팀별 합계와 일치한다.
  - 본부 행과 팀 행이 시각적으로 구분된다.

### Phase 4. 홈 가동률/가득률 본부 행 추가
- `TeamUtilization` 제목을 `가동률 / 가득률`로 변경한다.
- 기존 `rows`를 합산해 `PMO본부` 행을 상단에 추가한다.
- 계산:
  - 본부 인원 = 팀별 `headcount` 합계
  - 본부 가동률 = `(running + proposing) / total * 100`
  - 본부 가득률 = `running / total * 100`
- 브라우저 확인:
  - `PMO본부` 행이 테이블 상단에 표시된다.
  - 본부 가동률/가득률이 팀 합산 기준으로 계산된다.

### Phase 5. 업무수행현황 직접 기간 필터
- `ExecutionFilterState.periodPreset`에 `"custom"` 추가.
- `PeriodPicker`에 `직접 선택` 옵션과 `시작일`, `종료일` 입력을 추가한다.
- 조회 전에는 `filterForm`만 변경하고, 조회 시 `appliedFilter`에 반영하는 기존 패턴을 유지한다.
- 브라우저 확인:
  - 기존 프리셋이 기존처럼 작동한다.
  - 직접 선택 날짜 범위로 `제안서 제출일` 기준 필터링된다.

### Phase 6. 진행이력 직접 기간 필터
- `PeriodPreset`에 `"custom"` 추가.
- 기존 커스텀 드롭다운 `PeriodPicker`에 날짜 범위 입력을 추가한다.
- 조회 시 `row.datetime` 기준으로 필터링한다.
- 브라우저 확인:
  - 직접 선택 기간으로 진행이력 목록이 필터링된다.
  - 필터 요약에 기간이 표시된다.

### Phase 7. 프로젝트 관리 기간 필터
- `CodeFilterState`에 `periodPreset`, `from`, `to` 추가.
- 기본값은 `{ periodPreset: "all", from: "", to: "" }`.
- 필터 UI에 기간 항목을 추가한다.
- 프로젝트 관리 필터 레이아웃은 겹침 방지를 위해 `grid` 기반으로 재배치한다.
- `filteredRows`에 `row.fromDate` 기준 기간 조건을 추가한다.
- 브라우저 확인:
  - `사업 시작일` 기준 기간 필터가 작동한다.
  - 필터 항목과 버튼이 한 줄/두 줄 전환 시 겹치지 않는다.

### Phase 8. 업무수행현황 CSV 다운로드
- `filteredRows`를 엑셀(`.xlsx`)로 다운로드하는 공통 helper를 연결한다.
- 컬럼은 현재 사업 목록 테이블과 동일한 사용자 표시 컬럼을 기준으로 한다.
- 파일명은 `업무수행현황_YYMMDD_hhmm.xlsx`.
- 버튼은 사업 목록 헤더 우측의 필터 정보 오른쪽에 배치한다.
- 브라우저 확인:
  - 필터 적용 후 CSV 행 수가 검색결과 건수와 일치한다.
  - 한글이 Excel에서 깨지지 않는다.

### Phase 9. 프로젝트 관리 CSV 다운로드
- `filteredRows`를 엑셀(`.xlsx`)로 다운로드하는 공통 helper를 연결한다.
- 컬럼은 현재 프로젝트 목록 테이블의 표시 컬럼을 기준으로 한다.
- 파일명은 `프로젝트관리_YYMMDD_hhmm.xlsx`.
- 버튼은 프로젝트 목록 헤더 우측의 필터 정보 오른쪽에 배치한다.
- 브라우저 확인:
  - 페이지네이션과 무관하게 전체 필터 결과가 다운로드된다.
  - 파일명이 지정 형식과 일치한다.

### Phase 10. 제출 형식 이메일 옵션
- `ProjectMasterEditModal.tsx`의 `제출 형식` select에 `이메일` 옵션을 추가한다.
- `CodePage.tsx` 내부 모달의 `제출 형식` select에도 동일하게 추가한다.
- 브라우저 확인:
  - 신규 프로젝트 등록에서 `이메일`이 보인다.
  - 편집 모달에서도 `이메일`이 보인다.
  - 저장 후 목록/상세에 `이메일` 값이 표시된다.

### Phase 11. 업무수행현황 code 데이터 lazy load
- `ExecutionPage.tsx`의 페이지 진입 즉시 `getP1Screen("code")` 호출을 제거한다.
- `editingProject` 또는 `creatingProject`가 열릴 때 `codeRows`가 비어 있으면 한 번만 로드한다.
- 저장 후 refresh 흐름은 기존 동작을 유지한다.
- 브라우저 확인:
  - 업무수행현황 페이지 첫 진입 시 `/p1-screens/code` 호출이 없다.
  - 신규 등록/편집 모달을 열면 필요한 데이터가 로드되고 모달이 정상 작동한다.

### Phase 12. lightweight loading 적용
- 홈, 업무수행현황, 진행이력, 프로젝트 관리의 `return null` 초기 렌더를 간단한 로딩 패널로 교체한다.
- PmoShell을 사용할 수 없는 데이터 로딩 전에는 최소 높이와 텍스트만 있는 화면으로 처리한다.
- 브라우저 확인:
  - 페이지 전환 시 빈 화면이 아니라 로딩 상태가 보인다.
  - 로딩 후 기존 화면으로 자연스럽게 전환된다.

### Phase 13. API 성능 점검 및 필요한 범위만 개선
- `/api/p1-screens/home`, `/execution`, `/code`, `/history`의 쿼리 호출 패턴을 확인한다.
- N+1이 확인되는 함수만 bulk query로 개선한다.
- 우선 후보:
  - `_project_row()`의 `_person_name_with_title()` 반복 호출
  - 화면별 프로젝트/인력/팀 조회 반복 구간
- 브라우저 확인:
  - 데이터 값이 변경 전과 동일하다.
  - 페이지 진입 체감 시간이 개선된다.

### Phase 14. 최종 검증
- `npm run typecheck` 실행.
- 가능하면 `npm run build:web` 실행.
- 브라우저에서 홈, 업무수행현황, 진행이력, 프로젝트 관리, 프로젝트 등록/편집 모달을 순회 확인한다.

## 4. 완료 기준

- 홈에 `WIN`, `LOSS` 올해 누적 카드가 표시된다.
- 홈의 인력 현황/가동률 현황에 `PMO본부` 행이 추가된다.
- 기간 직접 선택 필터가 업무수행현황, 진행이력, 프로젝트 관리에서 동작한다.
- 업무수행현황/프로젝트 관리 엑셀 다운로드가 필터 결과 전체 기준으로 동작한다.
- 파일명이 `업무수행현황_YYMMDD_hhmm.xlsx`, `프로젝트관리_YYMMDD_hhmm.xlsx` 형식을 따른다.
- 제출 형식에 `이메일`이 추가된다.
- 업무수행현황 초기 진입 시 모달용 `code` 데이터 선로드가 제거된다.
- 각 phase는 브라우저 확인 완료 후 다음 phase로 넘어간다.
