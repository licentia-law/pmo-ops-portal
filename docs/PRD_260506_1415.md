## 1. 문서 개요

### 1.1 배경
현재 PMO본부는 `PMO본부 업무수행 현황_260331.xlsx`와 `2026년 PMO본부 KPI_가동율_260410.xlsx` 두 파일을 중심으로 업무를 운영하고 있다.

두 파일은 각각 다음 역할을 담당한다.

- `PMO본부 업무수행 현황_260331.xlsx`
  - 사업 운영 원장
  - 프로젝트/사업 일정 관리
  - PM/투입인력 메모 관리
  - 자유서술형 진행이력/특이사항 관리
- `2026년 PMO본부 KPI_가동율_260410.xlsx`
  - 인력 재직 MM 계산
  - 프로젝트 코드 관리
  - 인력×프로젝트×기간 기반 MM 계산
  - 기준일 현재 인원 스냅샷
  - 월별 가동률/가득률 집계
  - 보고용 리포트 시트 출력

### 1.2 이번 PRD의 핵심 방향
이번 버전의 PRD는 **이상적인 신규 PMO 시스템 설계 문서가 아니라, 기존 엑셀 2종을 브라우저로 재현하는 MVP 문서**다.

핵심 원칙은 다음과 같다.

1. **CEN.md는 완전히 무시한다.**
2. 변경 합의가 없는 항목은 **기존 엑셀 2종 기준 유지**를 우선한다.
3. MVP의 목적은 엑셀 운영 체계를 브라우저에서 입력, 조회, 집계, 다운로드 가능하게 만드는 것이다.
4. 데이터 구조는 브라우저 구현에 맞게 분리하되, **결과와 흐름은 기존 엑셀과 동일한 해석**을 우선한다.

### 1.3 제품 비전
기존 엑셀 2종의 운영 원장 + KPI 엔진 + 보고 템플릿 기능을 브라우저 안에서 통합 제공하여,

- 원장 입력
- 프로젝트 코드 관리
- 인력 재직/투입 관리
- 기준일 현재 인원 스냅샷
- 월별 MM 집계
- 가동률/가득률 산출
- 보고서 다운로드

를 하나의 내부 시스템에서 운영할 수 있도록 한다.

## 2. 목표와 범위

### 2.1 프로젝트 목표
이 시스템의 목표는 다음과 같다.

1. 기존 엑셀 2종의 핵심 기능과 산출물을 브라우저에서 재현
2. 사업 원장, 프로젝트 코드, 인력 재직 정보, 프로젝트 투입 정보, 진행 메모를 구조화하여 저장
3. 기존 KPI 파일의 기준일 스냅샷/월별 MM/월별 가동률·가득률 산식을 그대로 반영
4. 기존 보고 시트 성격의 화면과 엑셀 다운로드를 제공
5. 연차 제외, 외부접속, 고도화된 권한/감사 로그 등은 후속 확장 가능 구조로 설계

### 2.2 MVP 목표
MVP의 목표는 아래와 같다.

- 기존 엑셀 2종의 내용을 브라우저 안에서 조회/입력/수정 가능
- 기존 KPI 엔진의 핵심 계산 흐름을 재현 가능
- 기존 보고 시트 성격의 화면을 브라우저에서 확인 가능
- 기존 보고 산출물을 엑셀로 다운로드 가능
- 관리자/본부장/프로젝트 담당자 권한 흐름을 최소 수준으로 구현 가능

### 2.3 비목표 (MVP 제외)
아래 항목은 MVP 범위에서 제외하고 추후 확장으로 미룬다.

- 연차 사용 기간 자동 제외 계산
- 외부망 접속
- 실시간 알림
- Teams 연동
- PDF 출력 / 화면 인쇄 최적화
- 상세 감사 로그
- SSO/사내 계정 연동

## 3. 사용자와 권한

### 3.1 주요 사용자
- 본부장
- 팀장
- PM
- PL
- 조회 전용 사용자
- 관리자

### 3.2 로그인 방식
- **MVP는 로컬 계정 방식만 사용**한다.
- 사내 계정 연동, SSO, 외부 OAuth는 MVP 범위에서 제외한다.

### 3.3 데이터 접근 기본 원칙
- 기본적으로 조직 범위(본부/부문/팀)에 따라 접근 범위를 제한할 수 있어야 한다.
- MVP는 기존 운영 현실을 반영해 **관리자 중심 운영**을 우선한다.
- 세부적인 role/scope 세분화는 구현하되, 운영 흐름은 아래 3.4를 따른다.

### 3.4 운영 책임자 정책
#### 초기 정책
- 프로젝트 최초 등록: **관리자**
- 마스터 데이터 관리: **관리자 및 본부장**
- 월마감 책임자: **관리자 및 본부장**

#### 발표완료 이후 정책
- 해당 프로젝트가 `발표완료` 이후 상태로 넘어가면,
  - 해당 프로젝트의 **PM / 제안팀**이
  - 프로젝트 상세 내역과 투입기간을 수정할 수 있어야 한다.

### 3.5 권한 개념
MVP에서는 아래 수준을 최소 구현 단위로 둔다.

단, 실제 운영 우선순위는 아래가 더 중요하다.

- 최초 등록 권한은 관리자 중심
- 발표완료 이후 해당 프로젝트 담당자 수정 가능
- 마스터/월마감은 관리자 및 본부장 중심

## 4. 핵심 용어와 상태 체계

### 4.1 최종 상태값
최종 상태값은 아래 8개로 확정한다.

- 제안중
- 발표완료
- WIN
- LOSS
- DROP
- 수행중
- 업무지원
- 완료

### 4.2 상태값 의미
- `제안중`: 제안 준비/작성/접수 전후를 포함한 진행 중 상태
- `발표완료`: 발표까지 완료되고 결과 대기 또는 결과 확정 직전 상태
- `WIN`: 제안 결과 수주(제안 성공)
- `LOSS`: 제안 결과 실주(제안 실패)
- `DROP`: 제안/진행 중단
- `수행중`: 수주 후 실제 이행 수행 상태
- `업무지원`: 템플릿 작성 지원, 단기 수행 지원, 보조성 지원 업무 상태
- `완료`: 수행 또는 업무지원 종료 상태

### 4.3 상태 전환 규칙
상태 박스를 클릭했을 때는 **허용된 다음 흐름의 상태만 선택 가능**하도록 한다.

#### 기본 흐름
- `제안중 → 발표완료 / DROP`
- `발표완료 → WIN / LOSS`
- `WIN → 수행중`
- `수행중 → 완료`

#### 지원 흐름
- `업무지원 → 완료`

#### 완료 상태
다음 상태들은 종료 상태로 본다.

- WIN
- LOSS
- DROP
- 완료

> 해석 보완:
> - `WIN`은 제안 단계의 종료 상태이면서, 운영상 `수행중`으로 이어질 수 있는 연결점이다.
> - `LOSS`, `DROP`, `완료`는 최종 종료 상태다.

### 4.4 사업유형
사업유형 표준은 아래 4종으로 확정한다.

- 주사업
- 부사업
- 하도
- 협력

보조 설명:
- 기존에 사용되던 `들러리`, `모자` 개념은 시스템 용어로 사용하지 않는다.
- 참여 유형에 따라 `주사업 / 부사업 / 하도 / 협력` 안에서 해석한다.

## 5. 계산 정책

### 5.1 가동률 / 가득률 기본 원칙
가동률/가득률은 **기존 엑셀 공식 그대로 유지**한다.

### 5.2 주간현황 가동률 / 가득률
주간현황 시트 성격의 KPI는 **사람 수 기준 공식**으로 계산한다.

#### 원칙
- 주말/공휴일을 직접 차감해서 계산하지 않는다.
- 기준일 현재 사람을 어떤 상태로 보느냐를 기준으로 계산한다.

#### 현재 엑셀 철학
- 가동률 = `(계약/수행 + 제안) / 현재인원`
- 가득률 = `계약/수행 / 현재인원`

즉, 주간현황의 가동률/가득률은 **인원 카운트 기준 KPI**다.

### 5.3 월별 MM 계산
월별 MM 계산은 **기존 KPI 파일 방식 그대로** 유지한다.

#### 원칙
- 주말 제외
- 공휴일 제외
- 달력 시트 기준 공휴일 반영

즉, 월별 MM은 **근무 가능 일수 기준 계산**이다.

### 5.4 연차 제외 기능
- 기존 엑셀에는 연차 제외 기능이 없다.
- **MVP에서는 구현하지 않는다.**
- 추후 확장 기능으로 남겨둔다.

### 5.5 투입률(%) 정책
이번 MVP에서는 **배정별 투입률(%) 입력 기능을 도입하지 않는다.**

#### 이유
- 사용자가 “가동률과 같은 의미일 경우 투입률 관련 사항은 제외”로 확정했다.
- 기존 엑셀 재현형 MVP에서는 **투입률 입력보다 투입인력 + 투입기간 + 상태/구분**이 더 중요하다.
- KPI 파일의 MM 계산 철학은 유지하되, 브라우저 입력 항목으로 별도 `%` 필드를 강제하지 않는다.

#### MVP 입력 기준
프로젝트 인력 배정은 우선 아래 정보 중심으로 관리한다.

- 누가 투입되었는지
- 어떤 역할인지
- 제안/수행/지원 중 무엇인지
- 언제부터 언제까지인지
- 현재 기준으로 투입/미투입인지

## 6. 데이터 입력 정책

### 6.1 필수 입력값
필수 입력값은 아래와 같이 확정한다.

- 제안 PM
- 투입인력
- 사업명
- 프로젝트코드(자동입력)
- 사업금액
- 사업유형
- 시작 기준일(제안시작일)
- 지원 완료일(결과 발표일 또는 제출일)
- 주관부서(영업대표 입력 시 자동입력)
- 영업대표
- 사업지원접수일
- 제출일
- 발표일
- 발표 PM
- 상태값

### 6.2 선택 입력값
선택 입력값은 아래와 같이 둔다.

- 공고번호
- 공고일
- 세부 비고
- 진행 메모

### 6.3 프로젝트 코드 정책
프로젝트 코드는 **기존 방식 그대로 유지**한다.

- 프로젝트 신규 등록 시 시스템 자동 입력
- 필요 시 관리자 수기 수정 허용
- 코드 자릿수 체계도 MVP에서는 기존 데이터 호환을 우선한다.

### 6.4 날짜 입력 정책
날짜 입력은 **기본은 날짜, 필요 시 시간 포함** 방식으로 구현한다.

#### 기본 규칙
- 날짜 필드는 우선 날짜 선택형으로 입력
- 옆에 체크박스 `시간 포함`
- 체크 시 추가 시간 입력칸 노출

#### 적용 예시
- 제출일
- 제출시간(선택)
- 제출방식(현장/온라인)
- 제출비고
- 발표일
- 발표시간(선택)
- 발표방식/비고

#### 목적
- 기존 엑셀처럼 날짜, 시간, 방식, 설명이 한 셀에 뒤섞이는 문제를 줄이되
- 컬럼 수가 과도하게 많아지지 않도록 한다.

## 7. 기존 엑셀 재현 범위

### 7.1 MVP 구현 원칙
MVP는 **기존 엑셀 2개 파일의 내용 전부 포함**을 목표로 한다.

즉, 과거 PRD처럼 “최소 화면만 선별 구현”이 아니라,
**기존 엑셀에서 운영에 사용하던 내용은 전부 브라우저에서 구현 가능한 수준**을 목표로 삼는다.

### 7.2 기준 파일 1: PMO본부 업무수행 현황_260331.xlsx
이 파일에서 브라우저로 구현해야 할 핵심은 다음과 같다.

- 업무수행현황 원장
- 프로젝트별 기본 정보
- 사업 일정 정보
- PM/투입인력 정보
- 주관부서 / 영업대표 정보
- 진행 메모 / 기타사항
- 프로젝트코드 시트 성격의 간이 마스터

### 7.3 기준 파일 2: 2026년 PMO본부 KPI_가동율_260410.xlsx
이 파일에서 브라우저로 구현해야 할 핵심은 다음과 같다.

- 프로젝트코드 마스터
- 인력_투입현황MM

### 7.4 구현 우선순위 해석
“전부 포함”은 아래 두 층으로 나누어 해석한다.

1. **운영 핵심 엔진 반드시 구현**
   - 원장 입력
   - 인력/프로젝트 마스터
   - 기준일 스냅샷

## 11. 계산 플로우 구현 원칙

### 11.1 전체 플로우
브라우저 구현 시 계산 플로우는 기존 KPI 파일의 핵심 엔진 흐름을 따른다.

```text
공휴일(달력)
 +
인력재직현황
 +
프로젝트코드
 +
인력×프로젝트 투입현황
 ↓
기준일 현재 인원 스냅샷
대기현황 스냅샷
월별투입현황 집계
 ↓
주간현황 / 월별가동현황 / 대기·제안인원 / 제안PRJ / 이행PRJ
```

### 11.2 주간현황 KPI 계산
주간현황은 사람 수 기준으로 계산한다.

- 현재인원 = 가동계 + 비가동계
- 가동계 = 계약(수행) + 제안
- 비가동계 = 대기 + 사업부장 + 지원
- 가동률 = 가동계 / 현재인원
- 가득률 = 계약 / 현재인원

### 11.3 월별 MM 계산
월별 MM은 기존 KPI 파일 방식처럼 다음 기준을 따른다.

- 시작일/만료일과 해당 월의 겹치는 비율
- 월 근무가능일 대비 비율
- 근무가능일 = 주말 제외 + 공휴일 제외

### 11.4 대기 / 중복투입 정책
사용자 확정사항에 따라 **대기인원 / 중복투입 처리 기준은 기존 엑셀 그대로 유지**한다.

즉, MVP에서는 별도 철학 재설계보다 기존 엑셀의 해석을 재현하는 데 집중한다.

## 13. 비즈니스 규칙

### 13.1 상태 규칙
- 상태값은 제안중 / 발표완료 / WIN / LOSS / DROP / 수행중 / 업무지원 / 완료만 허용
- 상태 변경 UI에서는 허용된 다음 상태만 노출
- 업무지원 종료는 완료로 처리

### 13.2 권한 규칙
- 최초 프로젝트 등록은 관리자 중심
- 발표완료 이후 해당 프로젝트 PM/제안팀이 상세 정보와 투입기간 수정 가능
- 마스터/월마감은 관리자 및 본부장 중심

### 13.3 계산 규칙
- 주간현황 가동률/가득률은 사람 수 기준
- 월별 MM은 주말/공휴일 제외 기준
- 연차 제외는 MVP 미포함
- 기존 엑셀 공식 우선

### 13.4 데이터 재현 규칙
- 변경 합의가 없는 항목은 기존 엑셀 재현을 우선
- 구조화는 하되 결과는 기존 엑셀과 동일한 해석을 목표
- 보고 템플릿형 시트도 제외하지 않음

## 15. 비기능 요구사항 (NFR)

### 15.1 사용성
- 기존 엑셀 사용자가 큰 거부감 없이 적응할 수 있어야 함
- 원장 입력과 리포트 조회가 직관적이어야 함
- 날짜/시간 입력은 단순해야 함

### 15.2 정합성
- 기존 엑셀 기준 계산값과 비교 검증 가능해야 함
- 기준일 스냅샷과 월별 집계가 서로 모순되지 않아야 함
- 가동률/가득률 결과가 기존 엑셀 샘플과 일치해야 함

### 15.3 성능
- 수십 명 인력, 수십~수백 프로젝트 규모에서 무리 없이 동작
- 월간 집계와 리포트 다운로드가 실무 사용 가능한 속도로 동작

### 15.4 보안
- 비밀번호 해시 저장
- 권한별 접근 제어
- 로컬 계정 기반 최소 보안 준수

### 15.5 확장성
- 연차 제외 기능을 추후 붙일 수 있어야 함
- 외부 접속, 알림, Teams 연동, 감사 로그를 나중에 확장 가능해야 함

## 17. 리스크와 대응

### 17.1 기존 엑셀 수식 자체의 품질 이슈
- 리스크: KPI 파일 내부에 `#REF!`, 구식 상태/확도 로직, 수기 템플릿 의존이 존재
- 대응: 브라우저 구현 시 엑셀의 화면보다 **의도된 결과값과 흐름**을 기준으로 재구현

### 17.2 원장과 KPI 파일 간 코드 불일치
- 리스크: 운영 원장과 KPI 마스터 간 프로젝트코드가 완전히 일치하지 않음
- 대응: 브라우저에서는 통합 프로젝트 마스터를 두되, 기존 데이터는 이관 규칙으로 정리

### 17.3 투입률/연차에 대한 오해 가능성
- 리스크: MVP에서 투입률%와 연차 제외가 자동화된다고 오해할 수 있음
- 대응: PRD에 명시적으로 제외 범위를 기록

### 17.4 업무지원의 구조적 모호성
- 리스크: 업무지원이 상태인지 업무유형인지 장기적으로 애매할 수 있음
- 대응: MVP는 기존 엑셀 재현 우선, 이후 구조 고도화에서 재검토

## 18. 개발 원칙 (바이브코딩 전제)

- CEN.md는 완전히 무시한다.
- 기존 엑셀 2개 파일 재현이 최우선이다.
- 구조는 브라우저 친화적으로 분리하되 계산과 해석은 엑셀 기준을 따른다.
- 사용자가 명시적으로 바꾸자고 합의한 항목만 바꾼다.
- MVP에서는 과도한 이상화보다 실제 운영 재현을 우선한다.
- 결과 검증은 반드시 기존 엑셀 값과 비교한다.

## 19. 출시 판정 기준 (MVP DoD 초안)

아래 항목을 만족하면 MVP 운영 테스트 가능 상태로 본다.

1. 로컬 계정 로그인 동작
2. 업무수행현황 원장 등록/조회/수정 가능
3. 프로젝트코드 자동 생성 및 관리자 수정 가능
4. 발표완료 이후 PM/제안팀 상세 수정 권한 반영 가능
5. 인력재직현황 관리 가능
6. 인력×프로젝트 투입현황 관리 가능
7. 인원별_투입(현재) 기준일 스냅샷 구현 가능
8. 대기현황 전주/금주/차주 스냅샷 구현 가능
9. 월별 MM 계산이 주말/공휴일 제외 기준으로 구현 가능
10. 주간현황 가동률/가득률이 사람 수 기준 공식으로 일치
11. 월별가동현황/월별투입현황 결과가 샘플 검증과 일치
12. 보고 시트 성격 화면과 엑셀 다운로드 제공 가능
13. 연차 제외 기능이 MVP 범위에서 제외되었음이 명확히 분리됨

## 20. 최종 제안 요약

이 MVP는 새로운 PMO 포털을 설계하는 프로젝트라기보다,
**기존 엑셀 2종을 브라우저에서 운영 가능하게 다시 만드는 Excel Rebuild 프로젝트**로 정의하는 것이 맞다.

따라서 개발 순서는 다음을 권장한다.

1. 기존 엑셀 2종의 원장/마스터/팩트/리포트 구조를 기준으로 데이터 모델 확정
2. 원장 입력, 프로젝트코드, 인력재직현황, 인력×프로젝트 투입현황부터 구현
3. 기준일 현재 스냅샷과 월별 MM 집계 구현
4. 주간현황/월별가동현황/대기·제안인원/제안PRJ/이행PRJ 구현
5. 엑셀 다운로드와 운영 권한 흐름을 마감
6. 이후 연차 제외, 외부접속, 알림 등을 2차 확장
