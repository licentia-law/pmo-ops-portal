## 1. 문서 개요

### 1.1 배경

초기 PRD는 `PMO본부 업무수행 현황_260331.xlsx`와 `2026년 PMO본부 KPI_가동율_260410.xlsx` 두 엑셀 파일을 브라우저에서 재현하는 MVP를 목표로 작성되었다.

현재 프로젝트는 해당 방향을 유지하되, 구현은 다음 순서로 진행 중이다.

1. P1: 프로젝트 운영 원장, 프로젝트 코드, 프로젝트 상세, 진행이력, 홈 대시보드
2. P2: 인력 운영 화면/스냅샷/배정 원장 개편
3. P3 이후: 관리자 기준정보, 공휴일, 권한/로그인, 월마감
4. P4 이후: KPI 집계, 보고 화면, 다운로드 산출물 정교화

### 1.2 이번 PRD의 핵심 변경

이번 문서는 이상적인 최종 설계가 아니라, **현재 구현된 프로젝트를 기준으로 최신 요구사항을 재정렬한 기준 문서**다.

핵심 변경은 다음과 같다.

1. 현재 구현은 P1 프로젝트 운영 기능이 중심이다.
2. 홈, 업무수행현황, 프로젝트 관리, 프로젝트 상세, 진행이력은 실제 API와 연결되어 있다.
3. 인력/주간/월별 일부 화면은 아직 mock 또는 정적 화면 성격이 강하다.
4. P2 문서 기준으로 MVP 인력 메뉴는 `인력 투입 현황` 중심으로 개편해야 한다.
5. `대기/제안 현황`, `제안PRJ`, `이행PRJ`, `대기/제안인원`, `보고서 다운로드`의 별도 메뉴는 구현하지 않고 운영 화면 필터, 주간현황 상세 테이블, 화면별 다운로드 중심으로 정리한다.
6. 로컬 계정 로그인은 아직 실제 구현되지 않았고, 백엔드는 개발용 헤더 기반 사용자 주입으로 권한 흐름을 검증 중이다.

### 1.3 제품 비전

PMO본부의 프로젝트 운영 원장, 인력 투입 원장, 기준일 스냅샷, KPI 집계, 보고 다운로드를 하나의 내부 시스템으로 통합한다.

최종적으로 사용자는 다음 업무를 브라우저에서 수행할 수 있어야 한다.

- 프로젝트 등록/조회/수정
- 프로젝트 코드 및 마스터 관리
- 프로젝트 상세 일정/금액/PM/진행이력 관리
- 인력 재직/역할/배정 관리
- 기준일 현재 투입 현황 확인
- 상태 KPI 카드 필터를 통한 수행/제안/지원/대기 인력 목록 확인
- 월별 MM, 가동률, 가득률 집계 확인
- 보고용 엑셀 다운로드

## 3. 목표와 범위

### 3.1 제품 목표

1. 기존 엑셀 2종의 핵심 운영 흐름을 브라우저에서 재현한다.
2. 프로젝트 운영 원장과 프로젝트 코드를 DB 기준으로 구조화한다.
3. 진행이력을 별도 데이터로 저장하고 프로젝트 상세 및 전체 이력 화면에서 조회한다.
4. 인력 재직/배정/현재 투입 스냅샷을 DB 기준으로 관리한다.
5. 월별 MM, 주간 KPI, 월별 KPI를 백엔드 집계 결과로 제공한다.
6. 보고용 엑셀 다운로드를 화면별로 제공한다.

### 3.2 현재 MVP 우선순위

현재 MVP 우선순위는 다음 순서로 둔다.

| 우선순위 | 범위 | 상태 |
|---|---|---|
| P1 | 프로젝트 운영 원장/코드/상세/진행이력 | 구현 진행, 핵심 완료 |
| P2 | 인력 운영/대기·제안/배정 원장 | 요구사항 재정의 완료, 구현 필요 |
| P3 | 관리자/기준정보/공휴일/월마감 | 후속 |
| P4 | KPI 집계/주간·월별 보고 | 일부 모델 존재, 구현 필요 |
| P5 | 보고서 다운로드 | 프로젝트 화면 다운로드 일부 구현, 통합 다운로드 후속 |
| P6 | 로그인/권한 고도화/감사 로그 | 개발용 권한 검증만 존재 |

### 3.3 MVP 제외 또는 후속 범위

아래 항목은 현재 기준 MVP 후속 범위로 둔다.

- SSO/사내 계정 연동
- 외부망 접속
- Teams/메일 알림
- PDF 출력
- 화면 인쇄 최적화
- 상세 감사 로그
- 연차 사용 기간 자동 제외 계산
- 완전한 월마감 워크플로우
- 조직/팀 계층 기반 데이터 스코프 강제
- 별도 `대기/제안 현황` 페이지는 구현하지 않음. `주간현황` 상세 테이블로 대체

## 4. 사용자와 권한

### 4.1 주요 사용자

- 본부장
- 팀장
- PM
- PL
- 조회 전용 사용자
- 관리자

### 4.2 권한 코드

현재 백엔드 enum 기준 권한은 다음과 같다.

| 권한 | 코드 | 설명 |
|---|---|---|
| 조회 전용 | `read_only` | 조회만 가능 |
| 일반 수정 가능 사용자 | `general_editor` | 프로젝트 및 마스터 수정 가능 |
| 프로젝트 담당 수정 가능 사용자 | `project_editor` | 담당 프로젝트 조건부 수정 |
| 관리자 | `admin` | 전체 수정 가능 |

### 4.3 현재 인증 구현 상태

MVP 원칙은 로컬 계정 방식이지만, 현재 구현은 아직 실제 로그인/JWT가 아니다.

현재 API는 개발용 요청 헤더로 사용자를 주입한다.

- `x-user-name`
- `x-user-email`
- `x-user-permission`

헤더가 없으면 기본 관리자 사용자로 처리한다.

### 4.4 권한 정책

현재 구현된 정책은 다음과 같다.

1. `read_only`는 프로젝트/프로젝트 코드/진행이력 등록과 수정을 할 수 없다.
2. `admin`, `general_editor`는 프로젝트 등록/수정이 가능하다.
3. `project_editor`는 프로젝트가 발표완료 이후 상태이고, 본인이 해당 프로젝트 PM인 경우 수정할 수 있다.
4. 프로젝트 코드 마스터 수정은 현재 `admin`, `general_editor`만 가능하다.
5. 본부장 마스터 수정 허용 여부는 후속 정책 확정이 필요하다.

### 4.5 후속 구현 요구사항

- 로컬 계정 로그인 화면과 세션/JWT 구현
- 사용자 관리 화면 구현
- 조직/팀/본인 프로젝트 범위별 데이터 스코프 적용
- 본부장/팀장 권한 정책 확정
- 수정 이력 감사 로그 추가

### 5.2 상태 전환 규칙

현재 백엔드와 프론트 공통 규칙은 다음과 같다.

| 현재 상태 | 허용 다음 상태 |
|---|---|
| 제안중 | 발표완료, DROP |
| 발표완료 | WIN, LOSS |
| WIN | 수행중 |
| 수행중 | 완료 |
| 업무지원 | 완료 |
| LOSS | 없음 |
| DROP | 없음 |
| 완료 | 없음 |

동일 상태 저장은 허용한다. 허용되지 않는 상태 전환은 API에서 차단한다.

### 5.3 사업유형

사업유형 표준은 아래 4종이다.

| 라벨 | 코드 |
|---|---|
| 주사업 | `main` |
| 부사업 | `sub` |
| 하도 | `subcontract` |
| 협력 | `partner` |

### 5.4 배정 유형

현재 배정 유형은 다음과 같다.

| 라벨 | 코드 |
|---|---|
| 수행 | `delivery` |
| 제안 | `proposal` |
| 지원 | `support` |
| 미투입 | `unassigned` |

### 5.5 배정 상태

현재 배정 상태는 다음과 같다.

| 라벨 | 코드 |
|---|---|
| 예정 | `planned` |
| 배정 | `assigned` |
| 종료 | `ended` |

P2 최신 문서 기준으로는 `cancelled` 추가가 필요하다. 취소된 배정은 삭제하지 않고 이력으로 보존해야 한다.

### 5.6 진행이력 상태

진행이력 상태는 다음과 같다.

| 라벨 | 코드 | 정책 |
|---|---|---|
| 메모 | `memo` | 일반 기록 |
| 진행 | `in_progress` | 진행 중 조치 |
| 완료 | `done` | 진행 상태에서만 전환 가능 |

현재 구현 정책은 다음과 같다.

- 신규 등록 시 `done` 선택 불가
- `memo`와 `in_progress` 사이 직접 전환 불가
- `done`은 `in_progress`에서만 전환 가능
- `done` 상태 이력은 현재 수정 불가

## 8. 계산 정책

### 8.1 가동률/가득률

가동률/가득률은 기존 엑셀 철학을 유지한다.

주간현황은 사람 수 기준이다.

- 가동률 = `(수행 인원 + 제안 인원) / 현재 인원`
- 가득률 = `수행 인원 / 현재 인원`

월별가동현황은 MM 기준이다.

- 가동률 = `(수행 MM + 제안 MM) / 재직 MM`
- 가득률 = `수행 MM / 재직 MM`

### 8.2 월별 MM

월별 MM은 사람 × 프로젝트 × 월 기준으로 계산한다.

계산 원칙은 다음과 같다.

1. 월 근무 가능일을 산정한다.
2. 배정 기간과 해당 월의 겹치는 근무일을 산정한다.
3. 주말과 공휴일을 제외한다.
4. `배정 근무일 / 월 근무 가능일`을 기준으로 MM을 산정한다.
5. 확도 반영 MM이 필요한 경우 `certainty_rate`를 별도 weighted 값으로 저장한다.

### 8.3 연차 제외

연차 제외 기능은 기존 엑셀에 없으므로 MVP에서 제외한다.

추후 확장 시 연차/휴가 테이블을 추가하고, 월별 MM 계산에서 제외 기간으로 반영한다.

### 8.4 투입률 입력

MVP에서는 배정별 투입률 `%` 입력을 기본 필수값으로 도입하지 않는다.

현재 입력 중심은 다음이다.

- 인력
- 프로젝트
- 배정 유형
- 역할
- 시작일
- 종료일
- 상태
- MM 또는 월별 MM

## 11. 다운로드 요구사항

### 11.1 현재 구현

현재 프로젝트 화면 계열에는 브라우저 기반 엑셀 다운로드가 일부 구현되어 있다.

- 업무수행현황 다운로드
- 프로젝트 관리 다운로드

다운로드 파일명은 화면명과 현재 시각을 포함한다.

### 11.2 최신 요구사항

다운로드 정책은 다음과 같이 정리한다.

1. 우선 각 화면에서 현재 필터 결과를 다운로드한다.
2. `제안PRJ`, `이행PRJ`는 별도 페이지보다 업무수행현황 필터 + 다운로드로 대체한다.
3. 통합 `보고서 다운로드` 메뉴는 MVP 필수에서 제외하고 후속으로 둔다.
4. 서버 기준 산출물이 필요한 경우 report export API를 별도 구현한다.

## 12. 비기능 요구사항

### 12.1 사용성

- 화면은 기존 엑셀 사용자가 이해하기 쉬운 컬럼명과 흐름을 유지한다.
- 프로젝트 운영자는 목록에서 상세로 빠르게 진입할 수 있어야 한다.
- 본부장/팀장은 홈, 주간현황, 월별가동현황에서 핵심 KPI를 빠르게 확인할 수 있어야 한다.
- 관리자 화면은 입력 밀도가 높아도 필터/정렬/페이징이 안정적으로 동작해야 한다.

### 12.2 성능

- 목록 API는 페이징을 기본 지원한다.
- 대량 데이터 다운로드는 후속 서버 API로 분리할 수 있어야 한다.
- KPI 화면은 프론트에서 대량 계산하지 않고 백엔드 집계 또는 스냅샷 결과를 사용한다.

### 12.3 데이터 품질

- 프로젝트 코드 중복은 차단한다.
- 프로젝트 등록 필수값 누락은 API에서 차단한다.
- 허용되지 않는 상태 전환은 API에서 차단한다.
- 상태 변경은 자동 진행이력으로 남긴다.
- 배정 취소는 삭제가 아니라 상태 변경으로 보존한다.

### 12.4 보안

- MVP는 내부망/로컬 계정 기반을 전제로 한다.
- 개발용 헤더 인증은 운영 배포 전에 제거해야 한다.
- 운영 전 로그인, 세션, 권한 검증, CSRF/CORS 정책을 확정해야 한다.

## 15. 최종 정리

현재 프로젝트는 초기 PRD의 전체 MVP 중 **프로젝트 운영 P1 영역을 실제 API와 화면으로 구현한 상태**다.

새 기준선은 다음과 같다.

1. P1 프로젝트 운영 기능은 현재 구현을 기준으로 유지/보완한다.
2. 인력 영역은 P2 최신 문서 기준으로 `인력 투입 현황` 중심의 메뉴와 라우트로 개편한다.
3. KPI/보고 영역은 mock 화면에서 백엔드 집계/API 기반 화면으로 전환한다.
4. 권한은 현재 개발용 헤더 검증에서 로컬 계정 로그인 기반으로 전환한다.
5. 대기/제안 현황/제안PRJ/이행PRJ/대기·제안인원/보고서 다운로드 별도 메뉴는 구현하지 않고, 운영 화면 필터, 주간현황 상세 테이블, 화면별 다운로드 중심으로 재정리한다.

이 문서는 이후 구현/수정 작업의 최신 PRD 기준으로 사용한다.
