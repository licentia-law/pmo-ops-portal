## 1. 문서 개요

### 1.1 배경
현재 PMO본부는 업무수행 현황을 엑셀 중심으로 관리하고 있다. 그러나 실제 운영 방식은 단순 사업 목록 입력이 아니라 아래 업무가 동시에 혼합되어 있다.

- 사업 현황 원장 관리
- 프로젝트 코드 관리
- 인력 배정 관리
- 진행 메모/특이사항 관리
- 가동률/가득률 집계
- 월간 보고용 데이터 출력

이로 인해 데이터 구조가 비정형화되고, 집계/보고 시 수작업과 해석 차이가 누적될 수 있다.

### 1.2 문제 정의
현행 엑셀 기반 운영의 핵심 문제는 다음과 같다.

- 입력, 관리, 집계, 보고 역할이 하나의 파일에 혼재됨
- 상태값이 혼재되어 기준 해석이 흔들릴 수 있음
- 한 셀에 여러 명이 입력되는 등 구조화가 어려움
- 날짜/메모 항목이 비정형이라 자동 집계가 어려움
- KPI 산출 및 보고서 준비에 추가 수작업이 발생함

### 1.3 제품 비전
엑셀 기반의 업무수행 현황 관리를 브라우저 기반으로 전환하여, 인력/프로젝트/진행현황/통계/보고를 하나의 시스템 안에서 구조화하고 자동화한다.

## 2. 목표와 범위

### 2.1 프로젝트 목표
이 시스템의 목표는 다음과 같다.

1. 사업, 인력, 진행현황 정보를 브라우저에서 체계적으로 입력/관리
2. 엑셀에 섞여 있던 정보를 표준화된 구조로 분리/관리
3. 가동률, 가득률, 투입현황, 대기현황을 자동 집계
4. 월간 현황 및 보고용 엑셀 다운로드를 제공
5. 추후 권한 고도화, 알림, 월마감, 외부접속까지 확장 가능한 구조 마련

### 2.2 MVP 목표
MVP의 목표는 아래와 같다.

- 관리자/본부장이 사업과 인력을 브라우저에서 등록/수정 가능
- 프로젝트별 인력 배정 및 진행이력 기록 가능
- 월간 현황 대시보드와 기본 KPI 확인 가능
- 기본 보고서 데이터를 엑셀로 다운로드 가능
- 팀별 권한 기반 조회/수정 구조의 뼈대 확보

### 2.3 비목표 (MVP 제외)
아래 항목은 MVP 범위에서 제외하고 2차 이후로 미룬다.

- 외부망 접속
- 실시간 알림
- Teams 연동
- 수정이력 상세 감사 로그
- 주간 현황판
- PDF 출력 / 화면 인쇄 최적화
- SSO/사내 계정 연동

## 3. 사용자와 권한

### 3.1 주요 사용자
- 본부장
- 팀장
- PM
- PL
- 조회 전용 사용자
- 관리자

### 3.2 권한 레벨
권한은 아래 4단계로 정의한다.

1. **조회만 가능**
   - 본인 권한 범위 내 데이터 조회만 가능

2. **등록 가능**
   - 신규 등록 가능
   - 수정/삭제는 제한 가능

3. **수정 가능**
   - 등록 + 수정 가능

4. **관리자**
   - 전체 조회/등록/수정 가능
   - 계정/권한 관리 가능
   - 월마감 실행 가능

### 3.3 데이터 접근 범위
- 기본 정책: 팀별로 자기 팀 데이터만 조회
- 예외 정책: 관리자가 계정별 권한 변경을 통해 전체 조회 권한 부여 가능

### 3.4 운영 책임자 정책 (초기)
- 최초 사업 등록: 관리자, 본부장
- 진행이력 입력: 관리자, 본부장
- 월마감 실행: 관리자

### 3.5 운영 책임자 정책 (추후 확장)
- 최초 사업 등록: PM까지 이관 가능
- 진행이력 입력: PM 중심으로 이관 가능
- 월마감 실행: 본부장에게 이관 가능

## 5. 투입률 관리 정책

### 5.1 왜 필요한가
한 사람이 여러 프로젝트의 제안 작업을 동시에 하거나, 수행 중에 다른 제안 작업에 일부 참여할 수 있다. 따라서 단순히 “투입 여부만 표시”하는 방식으로는 현실 운영을 반영하기 어렵다.

### 5.2 투입률 관리 방식
**MVP는 배정별 투입률(%) 입력 방식으로 확정한다.**

#### 이유
- 한 사람이 여러 프로젝트의 제안/수행에 동시에 참여하는 현실을 가장 쉽게 반영할 수 있음
- 현업 사용자가 MM보다 직관적으로 이해하고 입력하기 쉬움
- 월간 집계와 엑셀 보고서 계산의 기반 데이터로 활용하기 좋음

### 5.3 입력 원칙
- 각 프로젝트-인력 배정 행마다 아래 정보를 저장한다.
  - 투입 시작일
  - 투입 종료일
  - 투입구분(제안/수행/지원)
  - 투입률(%)
  - 역할(PM/PL/참여 등)

### 5.4 운영 규칙
- 투입률은 배정 단위로 입력한다.
- 동일 기간 내 한 인력의 총합이 100%를 넘는 경우는 정책적으로 허용 여부를 추후 확정한다.
- MVP는 우선 저장 허용 + 경고 표시 방식을 기본안으로 둔다.
- MM 값은 사용자가 직접 입력하지 않고, 필요 시 시스템 계산 또는 보고서 파생값으로 생성하는 방향을 우선 적용한다.

### 5.4 KPI와의 관계
- 월간 현황/투입현황은 투입률 기반으로 집계 가능
- 단, 현재 엑셀의 “가동률/가득률”은 인원 카운트 성격이므로, 동시 투입자가 있을 경우 **대표 상태 분류 규칙**이 필요할 수 있음

### 5.5 대표 상태 분류
- 수행중 배정이 있으면: 수행 인원
- 수행은 없고 제안 배정이 있으면: 제안 인원
- 수행/제안은 없고 지원만 있으면: 지원 인원
- 아무 배정이 없으면: 대기 인원

## 6. 가동률 / 가득률 정책

### 6.1 결정사항
MVP는 **현재 엑셀 공식을 그대로 사용**한다.

### 6.2 현재 기준 공식
- 가동률 = (계약/수행 + 제안) / 현재인원
- 가득률 = 계약/수행 / 현재인원

### 6.3 해석
- 제안 인력은 가동으로 포함
- 계약/수행 인력은 가득으로 포함
- 지원/대기/기타 비가동 분류는 분모에 포함될 수 있음

### 6.4 주의사항
실제 시스템 설계 시 아래가 필요하다.
- 현재인원 정의 명확화
- 지원 인력 처리 방식 확인
- 팀장/본부장 등 관리 인력의 분모 포함 여부 확인

## 7. 로그인 방식 설명 및 권장안

### 7.1 가능한 로그인 방식

#### 방식 A. 간단 계정 방식 (로컬 계정)
- 시스템 내부에 아이디/비밀번호를 저장
- 가장 빠르고 구현이 단순
- 사내용 MVP에 적합

#### 방식 B. 사내 계정 연동 (AD/LDAP/SSO)
- 회사 계정으로 로그인
- 보안/운영 일원화에 유리
- 다만 초기 구현 복잡도와 사내 인프라 협의 필요

#### 방식 C. Microsoft/외부 OAuth 로그인
- Google/Microsoft 같은 외부 계정 로그인
- 일반 SaaS에는 مناسب하지만 사내 시스템에는 우선순위 낮음

### 7.2 확정안
**MVP는 간단 계정 방식(로컬 계정)**으로 확정한다.

#### 이유
- 사내 PC/사내 서버 환경에서 빠르게 구축 가능
- 역할/권한 테스트와 초기 운영 검증에 적합
- 추후 사내 계정 연동으로 확장 가능

### 7.3 보안 최소 기준
- 비밀번호 해시 저장
- 로그인 세션/토큰 만료 처리
- 계정별 역할(role)과 접근 범위(scope) 관리
- 초기 비밀번호 변경 기능 제공 권장

## 11. 주요 비즈니스 규칙

### 11.1 권한 규칙
- 기본은 팀 범위 내 데이터만 조회/수정 가능
- 관리자는 전체 조회/수정 가능
- 관리자가 개별 계정에 전체 조회 권한 부여 가능

### 11.2 프로젝트 등록 규칙
- 최초 등록자는 관리자/본부장
- 추후 PM 등록 이관 가능하도록 구조 설계

### 11.3 진행이력 규칙
- 프로젝트별 다건 이력 허용
- 최신 이력 우선 표시 가능
- 이력은 프로젝트 운영의 핵심 증적 데이터로 취급

### 11.4 투입률 규칙
- 한 인력은 여러 프로젝트에 동시 배정 가능
- 배정별 투입률을 입력해 월간 집계와 연결
- 총합 초과 정책은 경고 또는 제한 여부 추후 확정

### 11.5 KPI 규칙
- MVP는 기존 엑셀 공식을 우선 반영
- 세부 분류 규칙은 인터뷰 후 확정

## 13. 본부장 인터뷰 필요 항목

아래는 PRD 내 미확정 사항이며, 본부장 인터뷰에서 반드시 확인해야 한다.

### 13.1 첫 화면 KPI 우선순위
질문 예시:
- 첫 화면에서 가장 먼저 보고 싶은 숫자 5개는 무엇인가?
- 총원 / 수행 / 제안 / 대기 / 가동률 / 가득률 중 우선순위는?
- 프로젝트 수, 수주 수, 실주 수 등도 필요한가?

### 13.2 보고서 우선순위
질문 예시:
- 월간 보고에서 가장 먼저 필요한 출력물은 무엇인가?
- 인력 기준 보고와 프로젝트 기준 보고 중 어느 쪽이 더 중요한가?
- 기존 KPI 엑셀 구조를 유지할 필요가 있는가, 아니면 더 단순한 방식이 좋은가?

### 13.3 대표 상태 분류 규칙
질문 예시:
- 한 인력이 수행과 제안을 동시에 하면 어떤 상태로 대표 집계할 것인가?
- 지원 업무는 가동으로 볼 것인가, 비가동으로 볼 것인가?

### 13.4 월마감 필요성
질문 예시:
- 월말 기준 숫자를 확정해야 하는가?
- 지난달 보고 숫자가 이후 수정되면 안 되는가?

## 14. 기술 스택 제안

### 14.1 권장 기술 스택
#### 프론트엔드
- Next.js
- React
- TypeScript
- Tailwind CSS

#### 백엔드
- FastAPI
- SQLAlchemy 2.0
- Pydantic
- Alembic

#### 데이터베이스
- PostgreSQL

#### 인증
- MVP: 로컬 계정 + 세션 또는 JWT 기반 인증

#### 엑셀 출력
- Python 기반 엑셀 생성 라이브러리(openpyxl 또는 xlsxwriter 계열)

#### 배포
- 사내 PC 또는 사내 서버
- Docker Compose 선택 가능
- 초기에는 단일 서버 구성 권장

### 14.2 왜 이 구성을 추천하는가
- 프론트/백엔드 역할 분리가 명확해 유지보수에 유리
- 브라우저 UI와 API 확장성이 좋음
- 엑셀 다운로드, 권한, 대시보드 구현에 적합
- Streamlit보다 구조화된 권한/화면/운영 기능 구현에 유리
- 바이브코딩으로도 비교적 작업 분리가 쉬움

### 14.3 대안 스택
단순화가 더 필요하면 아래 대안도 가능하다.
- Next.js + FastAPI + SQLite (초기 로컬 검증 전용)

단, 실제 팀 사용을 고려하면 PostgreSQL로 가는 것이 안전하다.

## 15. 비기능 요구사항 (NFR)

### 15.1 사용성
- 비개발자도 쉽게 사용할 수 있어야 함
- 검색/필터/정렬이 직관적이어야 함
- 입력 폼은 과도하게 복잡하지 않아야 함

### 15.2 성능
- 일반 사내 사용 규모(수십~수백 프로젝트, 수십 명 사용자)에서 무리 없이 동작
- 목록 조회 및 대시보드 초기 로딩은 실무 사용 가능한 수준 유지

### 15.3 보안
- 비밀번호 해시 저장
- 역할 기반 접근 제어
- 팀 범위 제한
- 관리자 권한 분리

### 15.4 확장성
- 주간 현황판, 알림, 외부 접속, 감사 로그를 나중에 붙일 수 있어야 함
- 월마감/스냅샷 구조를 고려한 설계 필요

### 15.5 운영성
- 사내 PC/사내 서버에서 비교적 단순하게 운영 가능해야 함
- 백업/복구가 어렵지 않아야 함

## 17. 리스크와 대응

### 17.1 상태/집계 기준 미확정 리스크
- 리스크: 화면은 만들어도 숫자 해석이 달라질 수 있음
- 대응: 본부장 인터뷰로 기준 먼저 확정

### 17.2 투입률 규칙 미확정 리스크
- 리스크: 다중 프로젝트 참여 시 KPI 왜곡 가능
- 대응: 대표 상태 규칙과 초과 배정 정책 정의

### 17.3 권한 범위 복잡화 리스크
- 리스크: 팀별/전체 조회가 섞여 접근 제어가 복잡해짐
- 대응: role + scope 분리 구조 채택

### 17.4 엑셀 보고 요구 확장 리스크
- 리스크: 다운로드 양식 요구가 계속 늘어날 수 있음
- 대응: MVP는 핵심 보고서 1~2종부터 시작, 이후 확대

## 18. 개발 원칙 (바이브코딩 전제)

- 우선 기준 정의를 고정하고 구현 시작
- 화면보다 데이터 구조를 먼저 확정
- 엑셀과 100% 동일한 UI가 아니라 운영 목적 달성을 우선
- MVP는 단순하고 명확하게 유지
- 확장 기능(주간/알림/외부접속/감사로그)은 분리해서 단계적으로 진행
- 가동률/가득률 공식은 테스트 데이터로 검증 가능한 구조로 구현

## 19. 출시 판정 기준 (MVP DoD 초안)

아래 항목을 만족하면 MVP 운영 테스트 가능 상태로 본다.

1. 역할별 로그인 및 접근 제어 동작
2. 인력 마스터 등록/조회/수정 가능
3. 프로젝트 등록/조회/수정 가능
4. 프로젝트별 인력 배정 가능
5. 진행이력 등록/조회 가능
6. 월간 현황판에서 기본 KPI 확인 가능
7. 엑셀 보고서 다운로드 가능
8. 팀 범위 제한 및 관리자 전체 조회 권한 동작
9. 기존 엑셀 공식 기준 가동률/가득률 결과가 검증 샘플과 일치

## 20. 최종 제안 요약

이 시스템은 단순한 엑셀 입력 화면이 아니라, **프로젝트 관리 + 인력 운영 + KPI 집계 + 보고서 출력**을 결합한 내부 운영 시스템으로 정의하는 것이 맞다.

따라서 개발은 다음 순서를 권장한다.

1. 본부장 인터뷰로 KPI/보고/상태/투입률 기준 확정
2. MVP 범위에서 인력/프로젝트/배정/이력/월간현황/엑셀출력 구현
3. 이후 주간현황/알림/월마감/외부접속 등 2차 확장 진행
