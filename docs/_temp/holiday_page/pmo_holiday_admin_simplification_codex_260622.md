# PMO 업무시스템 공휴일 관리 기능 단순화 구현 요청

## 1. 작업 목적

PMO 업무시스템의 `관리 > 공휴일 관리` 화면과 관련 기능을 **복잡도 최소화** 방향으로 정리한다.

공휴일 관리는 사용자가 공휴일 데이터를 별도로 재가공하기 위한 기능이 아니라, 시스템 내 **근무일 계산, 월별 MM 계산, 가동률/가득률 산정의 기준정보**로 활용되는 기능이다. 따라서 법정공휴일은 외부 API 동기화로 자동 관리하고, 관리자는 회사휴무만 수동으로 관리하도록 단순화한다.

---

## 2. 현재 전제

현재 공휴일 관리 기능은 다음 구조를 가진다.

- 법정공휴일은 외부 공공데이터 API를 통해 동기화한다.
- 외부 API 동기화 데이터는 `source_kind = external_api`로 구분된다.
- 수동 등록 데이터는 `source_kind = manual`로 구분된다.
- 회사휴무는 수동 등록 대상이다.
- 공휴일 사용 여부는 근무일/MM 계산에 영향을 준다.
- 현재 화면에는 엑셀 다운로드, 반복여부, 비고, 사용여부 토글, 편집/삭제 기능이 노출되어 있다.

운영 정책은 아래처럼 정리한다.

| 구분 | 관리 방식 | 화면 조작 |
|---|---|---|
| 법정공휴일 | 외부 API 자동 동기화 | 조회만 가능 |
| 회사휴무 | 관리자 수동 등록 | 등록/수정/삭제/사용여부 변경 가능 |

---

## 3. 변경 방향 요약

### 핵심 방향

`공휴일 관리`는 “휴일 데이터 편집 화면”이 아니라 “근무일/MM 계산 기준 확인 및 회사휴무 관리 화면”으로 단순화한다.

### 최종 정책

#### 법정공휴일

- 외부 API 동기화로만 생성/갱신한다.
- 화면에서는 조회만 가능하게 한다.
- 사용여부 토글은 비활성화하거나 단순 상태 배지로 표시한다.
- 편집/삭제 버튼은 비활성화하거나 표시하지 않는다.
- 반복여부는 사용하지 않는다.
- 비고는 사용하지 않는다.

#### 회사휴무

- 관리자가 직접 등록한다.
- 사용여부 토글을 활성화한다.
- 편집/삭제를 허용한다.
- 반복여부는 사용하지 않는다.
- 비고는 사용하지 않는다.

## 4. 화면 변경 요청

### 4.1 엑셀 다운로드 기능 삭제

#### 대상

- 공휴일 목록 상단의 `엑셀 다운로드` 버튼
- 관련 프론트엔드 핸들러
- 관련 export 유틸이 이 화면에서만 사용된다면 해당 코드도 제거 검토

#### 사유

관리자가 공휴일 목록을 다운로드하여 별도로 재가공할 가능성이 낮다. 공휴일 데이터는 운영 기준정보이며, 다운로드 기능은 현재 MVP 범위에서는 불필요한 복잡도를 만든다.

#### 기대 결과

- 공휴일 목록 상단에서 `엑셀 다운로드` 버튼이 보이지 않는다.
- 버튼 제거로 인해 레이아웃이 어색하지 않아야 한다.

---

### 4.2 반복여부 기능 삭제

#### 대상

- 공휴일 목록 테이블의 `반복여부` 컬럼
- 신규 등록/수정 모달 또는 폼의 반복여부 입력 항목
- 프론트엔드에서 사용되는 `repeats_annually` 조작 UI

#### 사유

모든 휴일은 해당 연도 기준의 concrete row로 관리한다. 법정공휴일은 외부 API가 연도별 데이터를 제공하고, 회사휴무도 특정 연도 날짜 기준으로 등록하는 편이 MM 계산 정합성 측면에서 명확하다.

#### 기대 결과

- 목록 테이블에서 `반복여부` 컬럼이 사라진다.
- 신규 회사휴무 등록 시 반복여부를 입력하지 않는다.
- 프론트엔드에서 생성 요청을 보낼 때 `repeats_annually`는 항상 `false`로 처리하거나 백엔드 기본값을 사용한다.

---

### 4.3 사용여부 컬럼 정책 변경

#### 대상

- 공휴일 목록 테이블의 `사용여부` 컬럼
- 사용중 on/off 토글

#### 변경 정책

| 구분 | 사용여부 표시 방식 |
|---|---|
| 법정공휴일 | 토글 비활성화 또는 `자동관리`/`사용중` 배지 표시 |
| 회사휴무 | 토글 활성화 |

#### 사유

법정공휴일은 매일 새벽 1시경 외부 API로 갱신되는 자동관리 데이터다. 관리자가 임의로 사용여부를 변경해도 다음 동기화 시 다시 갱신될 수 있어 운영상 의미가 약하다.

반면 회사휴무는 내부 운영 기준에 따라 일시적으로 제외하거나 다시 활성화할 수 있으므로 사용여부 토글을 유지한다.

#### 기대 결과

- 법정공휴일 행에서는 사용여부를 직접 변경할 수 없다.
- 회사휴무 행에서만 사용여부 토글을 변경할 수 있다.
- 비활성화된 토글을 보여줄 경우 사용자가 혼동하지 않도록 tooltip 또는 배지 문구를 고려한다.
- 권장 표시: 법정공휴일은 토글 대신 `자동관리` 또는 `사용중` 배지 표시.

---

### 4.4 비고 컬럼 삭제

#### 대상

- 공휴일 목록 테이블의 `비고` 컬럼
- 신규 등록/수정 모달 또는 폼의 비고 입력 항목
- 프론트엔드에서 사용되는 `note` 조작 UI

#### 사유

법정공휴일과 회사휴무 모두 날짜, 명칭, 구분만으로 운영에 충분하다. 부가 설명을 기입할 일이 거의 없고, 입력 항목이 늘어나면 관리 복잡도만 증가한다.

#### 기대 결과

- 목록 테이블에서 `비고` 컬럼이 사라진다.
- 신규 회사휴무 등록/수정 폼에서 비고 입력 항목이 사라진다.
- DB 필드는 유지해도 되지만 화면에서는 사용하지 않는다.

---

### 4.5 작업 컬럼 정책 변경

#### 대상

- 공휴일 목록 테이블의 `작업` 컬럼
- 편집 버튼
- 삭제 버튼

#### 변경 정책

| 구분 | 편집 | 삭제 |
|---|---:|---:|
| 법정공휴일 | 불가 | 불가 |
| 회사휴무 | 가능 | 가능 |

#### 사유

법정공휴일은 외부 API로 자동 동기화되는 기준정보이므로 편집/삭제가 의미 없다. 수동으로 수정하더라도 다음 동기화 시 다시 덮어씌워질 수 있다.

회사휴무는 관리자가 직접 등록하는 데이터이므로 편집/삭제가 가능해야 한다.

#### 기대 결과

- 법정공휴일 행에서는 편집/삭제 버튼을 비활성화하거나 표시하지 않는다.
- 회사휴무 행에서만 편집/삭제 버튼을 표시한다.
- 권장 표시: 법정공휴일의 작업 컬럼에는 버튼 대신 `자동관리` 텍스트 또는 배지를 표시한다.

---

## 5. 버튼/문구 변경 요청

### 5.1 신규 등록 버튼명 변경

현재 버튼명:

- `신규 공휴일 등록`

변경 권장 버튼명:

- `회사휴무 등록`

#### 사유

관리자가 직접 등록해야 하는 대상은 법정공휴일이 아니라 회사휴무다. 버튼명에서 등록 대상을 명확히 하면 사용자의 혼동을 줄일 수 있다.

## 6. 백엔드 정책 보완 요청

프론트엔드에서 버튼을 숨기거나 비활성화하는 것만으로는 부족하다. 백엔드에서도 동일한 정책을 보장해야 한다.

### 6.1 법정공휴일 수정/삭제 제한 강화

현재 외부 API 동기화 공휴일에 대해서는 수정/삭제 제한이 있다. 운영 정책을 단순화하려면 아래 중 하나로 제한을 강화한다.

#### 권장안 A

`holiday_type == PUBLIC`인 데이터는 수정/삭제 불가 처리한다.

#### 권장안 B

`source_kind != MANUAL`인 데이터는 수정/삭제 불가 처리하고, `holiday_type == COMPANY`인 수동 데이터만 수정/삭제 허용한다.

#### 추천

권장안 B를 우선 추천한다.

- 외부 API 데이터: 수정/삭제 불가
- seed 데이터: 수정/삭제 불가 또는 별도 정리 대상
- manual 회사휴무: 수정/삭제 가능
- manual 법정공휴일: 신규 생성 자체를 막는 방향 권장

---

### 6.2 신규 등록은 회사휴무만 허용

공휴일 신규 등록 API 또는 프론트 요청에서 관리자가 직접 생성할 수 있는 구분은 회사휴무로 제한한다.

#### 정책

- 프론트엔드 등록 폼에서는 `구분` 선택값을 제거하거나 `회사휴무`로 고정한다.
- 백엔드에서도 수동 생성 시 `holiday_type == COMPANY`만 허용하는 방안을 검토한다.
- 법정공휴일은 API 동기화로만 생성한다.

---

### 6.3 사용여부 변경은 회사휴무만 허용

사용여부 변경도 회사휴무에 대해서만 허용한다.

#### 정책

- 법정공휴일의 `is_active` 변경 요청은 거부한다.
- 회사휴무의 `is_active` 변경 요청은 허용한다.
- 회사휴무 비활성화 시 `is_counted_as_workday = true`가 되어 근무일 계산에 포함되도록 기존 정책을 유지한다.

## 7. 데이터 정리 주의사항

### 7.1 기존 seed 법정공휴일 정리 필요

현재 구조에서는 과거 seed 데이터가 외부 API 동기화 후에도 남아 있을 수 있다. 예를 들어 실제 공휴일이 아닌 날짜가 seed 법정공휴일로 남아 있으면 근무일/MM 계산이 틀어질 수 있다.

따라서 법정공휴일을 UI에서 수정/삭제 불가로 만들기 전에 seed 데이터 정리 정책이 필요하다.

### 7.2 정리 권장 방식

아래 중 하나를 선택한다.

#### 방식 A: 외부 API 동기화 성공 연도에 대해 seed 법정공휴일 비활성화

- 특정 연도의 외부 API 동기화가 성공하면 해당 연도의 `source_kind = seed`, `holiday_type = public` 데이터를 비활성화한다.
- 단, 외부 API와 날짜/명칭이 명확히 매칭되는 데이터는 external_api 데이터로 전환하거나 중복 제거한다.

#### 방식 B: seed 법정공휴일을 수동 검토 후 삭제/비활성화

- 운영자가 2026년, 2027년 법정공휴일 목록을 확인한다.
- 외부 API 기준과 다른 seed 항목은 삭제 또는 비활성화한다.
- 이후 법정공휴일 수정/삭제 UI를 잠근다.

#### 추천

초기에는 방식 B를 추천한다. 현재 운영 초기 단계이므로 수동 검토 후 정리하는 것이 안전하다. 이후에는 방식 A처럼 자동 정리 정책을 추가할 수 있다.

## 9. 수용 기준

### UI 수용 기준

- [ ] 공휴일 목록 상단에서 `엑셀 다운로드` 버튼이 보이지 않는다.
- [ ] 공휴일 목록 테이블에서 `반복여부` 컬럼이 보이지 않는다.
- [ ] 공휴일 목록 테이블에서 `비고` 컬럼이 보이지 않는다.
- [ ] 법정공휴일 행에서는 사용여부를 직접 변경할 수 없다.
- [ ] 회사휴무 행에서만 사용여부 토글이 활성화된다.
- [ ] 법정공휴일 행에서는 편집/삭제가 불가능하다.
- [ ] 회사휴무 행에서만 편집/삭제가 가능하다.
- [ ] 신규 등록 버튼명이 `회사휴무 등록`으로 변경된다.
- [ ] 신규 등록 폼은 회사휴무 등록 목적에 맞게 단순화된다.

### 백엔드 수용 기준

- [ ] 수동 등록은 회사휴무만 허용하는 정책을 적용한다.
- [ ] 법정공휴일의 수정/삭제 요청은 백엔드에서 거부한다.
- [ ] 법정공휴일의 사용여부 변경 요청은 백엔드에서 거부한다.
- [ ] 회사휴무의 수정/삭제/사용여부 변경은 정상 동작한다.
- [ ] 비활성화된 회사휴무는 근무일 계산에서 휴일로 제외되지 않는다.
- [ ] 활성화된 회사휴무는 근무일 계산에서 휴일로 제외된다.

### 데이터 정합성 수용 기준

- [ ] 2026년/2027년 법정공휴일 목록에서 기존 seed 오류 데이터가 남아 있지 않다.
- [ ] 외부 API 동기화 후 법정공휴일 중복이 발생하지 않는다.
- [ ] 공휴일 기준 변경이 월별 근무일/MM 계산에 정상 반영된다.

## 10. 테스트 요청

### 프론트엔드 테스트

- 법정공휴일 행에서 사용여부 토글이 보이지 않거나 비활성화되어 있는지 확인한다.
- 법정공휴일 행에서 편집/삭제 버튼이 보이지 않거나 비활성화되어 있는지 확인한다.
- 회사휴무 행에서 사용여부 토글, 편집, 삭제가 정상 동작하는지 확인한다.
- 신규 등록 버튼명이 `회사휴무 등록`으로 표시되는지 확인한다.
- 신규 등록 폼에 반복여부/비고 입력 항목이 없는지 확인한다.

### 백엔드 테스트

- 수동 법정공휴일 생성 요청이 거부되는지 확인한다.
- 외부 API 법정공휴일 수정 요청이 거부되는지 확인한다.
- 외부 API 법정공휴일 삭제 요청이 거부되는지 확인한다.
- 외부 API 법정공휴일 사용여부 변경 요청이 거부되는지 확인한다.
- 회사휴무 생성/수정/삭제/사용여부 변경은 허용되는지 확인한다.
- 회사휴무 활성/비활성 상태에 따라 `business_days_between` 결과가 달라지는지 확인한다.

## 13. 최종 기대 효과

이번 변경으로 공휴일 관리 화면은 아래처럼 단순해진다.

- 관리자는 회사휴무만 관리한다.
- 법정공휴일은 자동 동기화 결과를 확인만 한다.
- 불필요한 엑셀 다운로드, 반복여부, 비고 입력을 제거한다.
- MM/근무일 계산 기준이 명확해진다.
- 운영 중 실수로 법정공휴일을 수정/삭제하는 위험이 줄어든다.
