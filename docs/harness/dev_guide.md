# 개발 진행 공통 가이드

이 문서는 PMO 업무수행 관리시스템 전체 개발 시 지켜야 할 공통 기준이다.  
특정 화면의 디자인 참고 파일이나 캡처본 목록은 `docs/harness/design_func_guide.md`를 따른다.

---

## 1. 핵심 원칙

### 1.1 복잡도 최소화

- MVP 범위에서는 문제를 가장 단순한 구조로 해결한다.
- 새 추상화는 실제 중복, 반복되는 규칙, 책임 분리가 확인될 때만 추가한다.
- 한 화면에서만 쓰는 로직은 먼저 화면 내부에 두고, 두 화면 이상에서 반복될 때 공통 모듈화를 검토한다.
- 드래그앤드롭, 고급 캘린더 편집, 자동 최적화, 과부하 자동 판정처럼 DTL에서 제외한 기능은 구현하지 않는다.
- 임시 우회 구현이 필요하면 TODO를 남기기보다 범위를 축소하거나 명시적으로 미구현 상태를 표시한다.

### 1.2 SSOT 준수

SSOT(Single Source of Truth)는 한 종류의 기준 데이터와 규칙을 한 곳에서 관리한다는 원칙이다.

- 라우트 경로는 `packages/shared-types/src/routes.ts`를 기준으로 한다.
- 사이드바 메뉴는 `packages/shared-types/src/menu.ts`를 기준으로 한다.
- enum/코드값은 백엔드 `apps/api/app/enums/__init__.py`와 프론트 shared types를 일치시킨다.
- API 응답 envelope는 `apps/api/app/api/common.py`의 `envelope()` 기준을 따른다.
- 같은 계산식은 프론트와 백엔드에 중복 구현하지 않는다. KPI, MM, 가동률/가득률 계산은 백엔드 기준으로 둔다.
- 화면 표시용 label/chip 매핑은 가능하면 공통 상수 또는 공통 helper로 모은다.
- DB/API에는 내부 enum value만 저장하고, 화면에는 공통 label helper를 통해 한글 라벨을 표시한다.
- DB에 `재직`, `수행`, `제안` 같은 화면 표시명을 직접 저장하지 않는다.

예시:

| 구분 | 내부값 | 화면 라벨 |
|---|---|---|
| 재직 상태 | active | 재직 |
| 재직 상태 | leave | 휴직 |
| 재직 상태 | transferred | 전배 |
| 재직 상태 | retired | 퇴직 |
| 배정 유형 | delivery | 수행 |
| 배정 유형 | proposal | 제안 |
| 배정 유형 | support | 지원 |
| 배정 유형 | unassigned | 대기 |

### 1.3 중복 기능은 공통 모듈화

두 개 이상의 화면에서 반복되는 기능은 공통 모듈화를 우선 검토한다.

- API 호출은 `apps/web/app/lib/api.ts`에 모은다.
- 기간/날짜 선택은 `apps/web/design/components/CommonPeriodPicker.tsx`를 우선 재사용한다.
- 로딩 표시는 `apps/web/design/components/LightweightLoading.tsx`를 우선 재사용한다.
- Shell/사이드바/헤더는 `apps/web/design/components/PmoShell.tsx`를 사용한다.
- workbook 다운로드는 기존 `apps/web/design/pages/projectWorkbookExport.ts` 패턴을 따르고, 인력용 다운로드가 반복되면 별도 공통 export helper로 분리한다.
- 백엔드 목록 API의 검색/정렬/페이지네이션은 `apply_text_search`, `parse_sort`, `paginate`를 재사용한다.

## 2. 프론트엔드 구조 기준

### 2.1 라우트와 화면 컴포넌트 분리

Next.js 라우트 파일은 얇게 유지한다.

```text
apps/web/app/(protected)/.../page.tsx
```

실제 화면 구현은 아래에 둔다.

```text
apps/web/design/pages/...
```

공통 컴포넌트는 아래에 둔다.

```text
apps/web/design/components/...
```

라우트 파일에는 metadata, search param 전달, 화면 컴포넌트 연결 정도만 둔다.

### 2.2 메뉴와 현재 화면 표시

- 메뉴 변경 시 `routes.ts`와 `menu.ts`를 함께 수정한다.
- 숨김 상세 라우트는 `menu.ts`에 추가하지 않는다.
- `PmoShell`의 `currentId`는 `menu.ts`의 item id와 맞춘다.
- 관리자 메뉴에만 노출되는 화면은 일반 업무 메뉴에 중복 노출하지 않는다.
- URL이 업무 도메인 경로이더라도 메뉴 노출과 권한 기준이 관리자 화면이면 관리자 화면으로 취급한다.

### 2.3 MVP 최종 메뉴 기준

MVP에서 사이드바에 노출하는 메뉴는 아래 구조를 기준으로 한다.  
이 범위를 벗어나는 보고서/상세/원장성 페이지는 DTL에서 명시하지 않는 한 메뉴에 추가하지 않는다.

```text
홈
└─ 홈

프로젝트
├─ 업무수행현황
└─ 진행이력

인력
├─ 인력 투입 현황
└─ 대기/제안 현황

KPI/보고
├─ 주간현황
└─ 월별가동현황

관리
├─ 사용자/권한 관리
├─ 기준정보 관리
├─ 프로젝트 관리
├─ 인력 관리
├─ 공휴일 관리
└─ 월마감/스냅샷

숨김 라우트
├─ 프로젝트 상세
└─ 인력별 투입 상세
```

`보고서 다운로드` 별도 페이지는 MVP에서 구현하지 않는다.  
각 화면의 우측 상단 엑셀 다운로드 버튼으로 대체한다.  
다운로드는 현재 화면에 적용된 필터 조건과 동일해야 한다.  
공통 export helper는 허용하지만 별도 `/reports/download` 화면은 만들지 않는다.

### 2.4 상태 관리

- 화면 내부 상태는 필터 입력값과 실제 적용값을 분리한다.
- `조회` 버튼은 적용값을 갱신하고 데이터를 재조회한다.
- `초기화` 버튼은 필터 기본값으로 되돌린다.
- 저장/수정 성공 후에는 관련 목록 또는 화면용 API를 재조회한다.

### 2.5 기준 시점 용어

인력/KPI 화면의 query와 필터명은 아래 용어를 기준으로 통일한다.

| 이름 | 의미 | 주요 사용 화면 |
|---|---|---|
| basis_date | 기준일. 현재 스냅샷 산정 기준 | 인력 투입 현황, 인력별 투입 상세 |
| basis_week | 기준주. 주간 운영판과 주간 KPI 산정 기준 | 대기/제안 현황, 주간현황 |
| basis_month | 기준월. 월별 MM과 월별 KPI 산정 기준 | 월별가동현황, 월별 MM |
| basis_year | 기준연도. 연간 재직 MM과 월별 추이 산정 기준 | 인력 관리, 월별가동현황 |

화면 라벨은 `기준일`, `기준주`, `기준월`, `기준연도`로 표시하되 API query와 내부 변수명은 위 이름을 우선 사용한다.

## 3. 백엔드/API 기준

### 3.1 API 응답 형식

모든 신규 API는 아래 envelope 형식을 따른다.

```json
{
  "data": {},
  "meta": {},
  "error": null
}
```

오류 응답도 `error_envelope()` 형식을 따른다.

### 3.2 목록 API

목록 API는 가능한 한 아래 query를 지원한다.

- `page`
- `page_size`
- `sort`
- `q`
- 상태/유형 등 화면별 필터

응답 `meta`에는 가능한 한 아래 값을 포함한다.

- `page`
- `page_size`
- `total`

### 3.3 화면용 API와 CRUD API 분리

- CRUD API는 단일 리소스의 조회/등록/수정 책임을 가진다.
- 화면용 API는 KPI, 필터 옵션, 대표 행, 보조 패널 데이터를 한 화면에서 쓰기 좋게 조합한다.
- 화면용 API에서 계산한 값이 저장 대상이 아니라면 별도 테이블을 만들지 않는다.
- 저장해야 하는 스냅샷/마감 데이터만 별도 집계 테이블을 둔다.

화면용 API 응답은 아래 형태를 권장한다. 모든 필드를 강제하지는 않지만, 같은 성격의 데이터는 같은 이름을 우선 사용한다.

```json
{
  "data": {
    "summary": {},
    "filters": {},
    "rows": [],
    "sidePanels": {},
    "downloadMeta": {}
  },
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 0
  },
  "error": null
}
```

### 3.4 계산 위치

- KPI, MM, 가동률/가득률, 중복 배정 판단은 백엔드에서 계산한다.
- 프론트는 백엔드 결과를 표시하고 필터링/정렬 UI를 제공한다.
- 프론트에서 임시 계산이 필요하면 화면 표시용으로만 제한하고, 저장/보고 기준으로 사용하지 않는다.

## 4. 검증과 완료 기준

### 4.1 구현 전

- 관련 DTL의 확정 라우트와 API 목록을 확인한다.
- 기존 구현에서 재사용 가능한 컴포넌트와 helper를 먼저 찾는다.
- 새 파일을 만들기 전에 같은 책임의 파일이 이미 있는지 확인한다.

### 4.2 구현 중

- mock 데이터는 초기 레이아웃 확인에만 사용한다.
- 완료 기준은 실제 API 연결과 seed DB 기준 동작이다.
- 필수값 검증은 프론트와 백엔드 모두에서 수행한다.
- 권한 또는 메뉴 노출 조건이 있는 화면은 라우트와 메뉴를 함께 점검한다.
- 관리자 화면은 메뉴 노출뿐 아니라 라우트 접근과 수정 API 권한도 제한한다.

### 4.3 구현 후

- 목록 조회, 필터, 등록, 수정, 상세 이동, 다운로드를 화면별로 확인한다.
- `npm.cmd run build` 또는 해당 프로젝트에서 요구하는 검증 명령을 실행한다.
- 문서에서 정한 제외 범위가 구현에 섞이지 않았는지 확인한다.
- 신규 공통 기능이 생겼다면 중복 화면이 같은 모듈을 쓰도록 정리한다.
- seed DB 기준으로 KPI 카드, 목록 행 수, 상태별 요약 숫자가 서로 일치하는지 확인한다.

### 4.4 인력 파트 숫자 검증

인력 투입 현황 검증:

- KPI 카드 합계와 현재 인원 스냅샷 행 수가 맞는지 확인한다.
- 수행/제안/지원/대기 수가 상태별 요약과 일치하는지 확인한다.
- 특정 인력 상세의 현재 배정과 현재 스냅샷의 대표 상태/프로젝트가 일치하는지 확인한다.
- 차기 투입 예정 목록과 배정 시작일 기준 결과가 일치하는지 확인한다.

대기/제안 현황 검증:

- 대기 인원 리스트와 KPI 대기 인원 수가 일치하는지 확인한다.
- 제안/수행 리스트와 `project_assignments.assignment_type` 기준 결과가 일치하는지 확인한다.
- 전주/금주/차주 운영판의 상태가 `weekly_load_snapshots` 기준 결과와 일치하는지 확인한다.

### 4.5 권한 검증

- `관리 > 인력 관리`는 메뉴 노출뿐 아니라 라우트 접근과 API 수정 권한도 admin 기준으로 제한한다.
- 일반 사용자가 관리자 URL을 직접 입력하면 접근 차단 또는 권한 없음 화면을 표시한다.
- read only 사용자는 등록/수정/상태 변경 버튼을 사용할 수 없어야 한다.
- 백엔드 API는 프론트 버튼 노출 여부와 관계없이 권한을 다시 검증한다.

## 5. 인력 파트 적용 메모

인력 파트 개발 시 특히 아래를 지킨다.

- `관리 > 인력 관리`는 `/people/employment` 라우트를 유지하되 관리 메뉴에만 노출한다.
- `/people/employment`는 URL은 people 하위이지만 관리자 화면으로 취급한다.
- `/people/employment`는 일반 인력 메뉴에 노출하지 않는다.
- `/people/employment`의 라우트 접근과 수정 API는 admin 권한 기준으로 제한한다.
- `인력 > 인력 투입 현황`은 `/people/current-assignments`로 구현한다.
- `인력 > 대기/제안 현황`은 `/people/waiting-proposals`로 구현한다.
- `인력별 투입 상세`는 `/people/assignments/[personnelId]` 숨김 라우트로 구현한다.
- 숨김 상세 라우트는 메뉴에는 노출하지 않지만 직접 URL 접근과 새로고침은 가능해야 한다.
- 상세 화면에는 목록으로 버튼을 둔다.
- 가능하면 목록 복귀 시 기존 필터 조건을 유지한다.
- `roles`는 신규 테이블/API로 구현한다.
- `AssignmentStatus.cancelled`, `WorkLocationType` enum은 백엔드/프론트 타입을 함께 맞춘다.
- `weekly-kpi-summaries`는 API는 제공하되 MVP에서는 스냅샷 기반 파생 응답으로 구현한다.
- `보고서 다운로드` 별도 페이지는 구현하지 않는다.
