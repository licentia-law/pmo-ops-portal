# 검증 가이드

스크립트 검증, 인력 숫자 검증, 권한 검증, seed DB 기준 검증을 정리한다.  
Codex는 검증 범위를 정하거나 완료 보고를 작성할 때 이 문서를 참고한다.

---

## 1. Codex 검증 범위

- Codex는 코드 변경 후 스크립트 검증까지만 수행한다.
- 브라우저에서의 화면 확인은 사용자가 직접 수행한다.
- 문서 변경만 한 경우에는 명령 실행 없이 변경 위치 확인으로 마무리해도 된다.
- 검증 실패 시 실패 원인과 미실행 항목을 최종 응답에 명확히 남긴다.

기본 명령:

```powershell
npm.cmd run typecheck
npm.cmd run build:web
```

## 2. 구현 전 체크

- 관련 DTL의 확정 라우트와 API 목록을 확인한다.
- 기존 구현에서 재사용 가능한 컴포넌트와 helper를 먼저 찾는다.
- 새 파일을 만들기 전에 같은 책임의 파일이 이미 있는지 확인한다.
- 메뉴/라우트 변경이면 `routes.ts`, `menu.ts`, `PmoShell currentId`를 함께 확인한다.

## 3. 구현 중 체크

- mock 데이터는 초기 레이아웃 확인에만 사용한다.
- 완료 기준은 실제 API 연결과 seed DB 기준 동작이다.
- 필수값 검증은 프론트와 백엔드 모두에서 수행한다.
- 권한 또는 메뉴 노출 조건이 있는 화면은 라우트와 메뉴를 함께 점검한다.
- 관리자 화면은 메뉴 노출뿐 아니라 라우트 접근과 수정 API 권한도 제한한다.

## 4. 구현 후 공통 체크

- 목록 조회, 필터, 등록, 수정, 상세 이동, 다운로드를 화면별로 확인한다.
- `npm.cmd run typecheck`를 실행한다.
- 필요 시 `npm.cmd run build:web`를 실행한다.
- 문서에서 정한 제외 범위가 구현에 섞이지 않았는지 확인한다.
- 신규 공통 기능이 생겼다면 중복 화면이 같은 모듈을 쓰도록 정리한다.
- seed DB 기준으로 KPI 카드, 목록 행 수, 상태별 요약 숫자가 서로 일치하는지 확인한다.
- 브라우저 화면 확인은 사용자가 직접 수행한다.

## 5. 삭제/정리 체크

- 라우트 삭제 시 `routes.ts`, `menu.ts`, page 파일, legacy 파일을 함께 확인한다.
- enum 삭제 시 백엔드 enum과 프론트 shared types를 함께 확인한다.
- 메뉴 삭제 시 관련 컴포넌트, mock, 디자인 소스, 문자열 참조를 `rg`로 확인한다.
- 보고서/다운로드 별도 페이지가 되살아나지 않았는지 확인한다.

## 6. 인력 투입 현황 숫자 검증

- KPI 카드 합계와 현재 인원 스냅샷 행 수가 맞는지 확인한다.
- 수행/제안/지원/대기 수가 상태별 요약과 일치하는지 확인한다.
- 특정 인력 상세의 현재 배정과 현재 스냅샷의 대표 상태/프로젝트가 일치하는지 확인한다.
- 차기 투입 예정 목록과 배정 시작일 기준 결과가 일치하는지 확인한다.

## 7. 권한 검증

- `관리 > 인력 관리`는 메뉴 노출뿐 아니라 라우트 접근과 API 수정 권한도 admin 기준으로 제한한다.
- 일반 사용자가 관리자 URL을 직접 입력하면 접근 차단 또는 권한 없음 화면을 표시한다.
- read only 사용자는 등록/수정/상태 변경 버튼을 사용할 수 없어야 한다.
- 백엔드 API는 프론트 버튼 노출 여부와 관계없이 권한을 다시 검증한다.

## 8. 완료 보고

- 변경 파일과 핵심 변경 내용을 짧게 요약한다.
- 실행한 검증 명령과 결과를 적는다.
- 실행하지 못한 검증이 있으면 이유를 적는다.
- 브라우저 확인은 검증 범위에 포함하지 않았다는 사실만 필요 시 말한다.
