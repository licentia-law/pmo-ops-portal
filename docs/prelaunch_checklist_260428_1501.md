- 사용 대상: 본인, 부서장 인터뷰, 바이브코딩 개발 착수 전 기준 정리

## 0. 이 문서의 목적

이 문서는 **개발 착수 전에 먼저 확정해야 하는 운영 기준**을 정리한 체크리스트다.

이 프로젝트는 단순히 엑셀 입력 화면을 브라우저로 옮기는 작업이 아니라,
다음 요소가 함께 얽혀 있다.

- 프로젝트 관리
- 인력 운영
- 인력 배정
- 진행이력 관리
- 가동률/가득률 집계
- 월간 현황 및 보고서 출력

따라서 화면 구현 전에 아래 항목을 먼저 확정해야,
이후 데이터 구조, 화면 설계, 집계 로직, 엑셀 다운로드 양식이 흔들리지 않는다.

## 1. 우선 확정해야 할 핵심 항목

### 1-1. 상태값 표준

#### 왜 먼저 정해야 하나
상태값은 프로젝트 목록, 대시보드, 월간 집계, 보고서 출력, 권한별 조회 결과에 모두 영향을 준다.
지금처럼 같은 의미의 값이 여러 방식으로 섞여 있으면, 같은 프로젝트도 사람마다 해석이 달라질 수 있다.

#### 기존 엑셀에서 보이는 목록
- `업무수행현황` 시트 `단계` / `PJT결과/진행상황` 계열
  - `제안진행`
  - `WIN`
  - `LOSS`
  - `DROP`
  - `수행중`
  - `수행중(WIN)`
  - `완료`
  - `업무지원`
- `프로젝트코드` 시트 `상태`
  - `제안중`
  - `지원중`
  - `BMT중`
  - `수주`
  - `실주`
  - `Drop`

#### 제안 목록
- `제안준비`
- `제안중`
- `발표완료`
- `수주`
- `실주`
- `수행중`
- `완료`
- `중단`

#### 부서장 확인 포인트
- `WIN`, `수주`, `수행중(WIN)`을 어떻게 정리할지
- `업무지원`, `BMT중`, `지원중`은 상태값인지 사업유형/세부단계인지
- 종료 상태를 `완료 / 실주 / 중단`으로 구분할지

---

### 1-2. 사업유형 표준

#### 왜 먼저 정해야 하나
사업유형은 프로젝트 성격을 구분하는 값이고, 상태값과 섞이면 집계가 꼬인다.
즉, “현재 어떤 단계인가”와 “어떤 유형의 사업인가”는 분리해서 관리해야 한다.

#### 기존 엑셀에서 보이는 목록
- `업무수행현황` 시트 `사업유형`
  - 주로 `주사업`이 확인됨
- 다만 상태/메모 영역에 `업무지원`이 섞여 있어
  - 실제 운영상 사업유형과 상태가 혼용된 것으로 보임

#### 제안 목록
- `주사업`
- `부사업`
- `협력참여`

#### 부서장 확인 포인트
- 위 3개로 시작해도 되는지
- `업무지원`을 사업유형으로 둘지, 별도 투입구분으로 둘지
- 향후 `유지보수`, `내부과제`, `지원` 같은 유형 추가 가능성이 있는지

---

### 1-3. 가동률 / 가득률 공식

#### 왜 먼저 정해야 하나
이 수치는 대시보드, 월간 현황판, 보고서의 핵심 KPI다.
공식이 조금만 달라도 보고 숫자가 달라지므로 반드시 개발 전 합의가 필요하다.

#### 기존 엑셀에서 보이는 방식
- `Ⅱ.1.주간현황`
  - 가동률 = `(계약 + 제안) / 현재인원`
  - 가득률 = `계약 / 현재인원`
- `월별투입현황`
  - 가동률 = `(수행 + 제안) / (수행 + 제안 + 지원 + 미투입)`
  - 가득률 = `수행 / (수행 + 제안 + 지원 + 미투입)`

#### 현재 제안
- MVP는 **기존 엑셀 공식 그대로 사용**
- 해석 기준만 명확히 문서화
  - 가동률 = `수행 + 제안` 포함
  - 가득률 = `수행만` 포함

#### 부서장 확인 포인트
- `지원` 인력을 분모에 넣고 분자에서 제외할지
- `본부장/팀장/관리직`을 분모에 포함할지
- 월간 집계와 현재 시점 집계를 같은 철학으로 볼지

---

### 1-4. 권한 구조

#### 왜 먼저 정해야 하나
이 프로젝트는 “누가 볼 수 있고, 누가 수정할 수 있는지”가 중요하다.
권한 구조를 초기에 정하지 않으면, 이후 화면/쿼리/API를 다시 뜯어고쳐야 한다.

#### 기존 엑셀 운영 방식
- 사실상 파일 공유 기반
- 엑셀 자체에는 명확한 역할/권한 체계 없음

#### 제안 목록
- 권한 레벨
  - `조회만 가능`
  - `등록 가능`
  - `수정 가능`
  - `관리자`
- 범위 정책
  - 기본은 **팀별 자기 팀 데이터만 조회**
  - 관리자가 계정별로 **전체 조회 권한 부여 가능**

#### 부서장 확인 포인트
- 본부장은 전체 조회만 하고 수정은 제한할지
- PM/PL에게 어느 시점부터 등록/수정 권한을 줄지
- 사용자 관리 화면은 관리자 전용으로 고정할지

---

### 1-5. 투입률 입력 기준

#### 왜 먼저 정해야 하나
한 사람이 여러 프로젝트의 제안 또는 수행에 동시에 일부 참여할 수 있기 때문에,
단순 투입 여부만으로는 실제 운영을 반영하기 어렵다.

#### 기존 엑셀에서 보이는 방식
- `업무수행현황`
  - 한 셀에 여러 명이 줄바꿈으로 입력됨
- `인력_투입현황MM`
  - 성명, 제안/수행, 상주, 계약기간, 월별 MM 구조 존재
- 즉, 원장은 비정형, KPI는 별도 시트에서 다시 구조화하는 형태

#### 제안 목록
- **배정별 투입률(%) 입력 방식**
- 1행 = `1명 × 1프로젝트 × 1구분 × 1기간`
- 예시
  - 김철수 / A프로젝트 / 수행 / 70%
  - 김철수 / B프로젝트 / 제안 / 30%

#### 부서장 확인 포인트
- 총합 100% 초과를 저장 불가로 할지, 경고만 할지
- `지원`도 동일하게 투입률 관리할지
- MM은 직접 입력하지 않고 시스템 계산값으로 둘지

---

## 2. 함께 정해두면 좋은 데이터 기준 항목

### 2-1. 필수 입력값 / 선택 입력값

#### 왜 먼저 정해야 하나
필수값이 너무 많으면 입력이 불편하고, 너무 적으면 집계 정확도가 떨어진다.
따라서 MVP에서는 최소 필수값을 정하고 나머지는 보완 입력하도록 설계하는 것이 좋다.

#### 기존 엑셀에서 보이는 주요 항목
- 팀명
- 정/부 직급, 성명
- 사업명
- 프로젝트코드
- 사업금액
- 사업유형
- 단계
- 지원시작일 / 지원완료일
- 사업 주관 부서
- 담당자
- 사업지원접수일
- 사업공고번호
- 사업공고일
- 제안제출일
- 제안발표일
- 발표/이행 PM
- 결과/진행상황
- 기타사항

#### 제안 목록
- **필수**
  - 사업명
  - 프로젝트코드
  - 사업유형
  - 상태값
  - 주관부서
  - 영업대표
  - 담당 PM
  - 기준 시작일(등록일 또는 지원시작일)
- **선택**
  - 사업금액
  - 공고번호
  - 공고일
  - 제출일
  - 발표일
  - 지원완료일
  - 비고
  - 진행 메모

#### 부서장 확인 포인트
- 실제 운영상 반드시 처음부터 입력해야 하는 항목이 무엇인지
- 금액/공고번호/발표일 같은 값이 초기 등록 시 꼭 필요한지

---

### 2-2. 프로젝트 코드 정책

#### 왜 먼저 정해야 하나
프로젝트 코드는 검색, 정렬, 보고서 연결, 인력 배정 참조의 기준 키 역할을 한다.
형식이 흔들리면 이후 데이터 정합성이 약해진다.

#### 기존 엑셀에서 보이는 목록
- `P202400016`
- `P202500006`
- `P2026001`
- `P2026011`
- 연도는 포함되나 자리수 길이가 일관되지 않음

#### 제안 목록
- 신규 정책: `P + YYYY + 4자리 일련번호`
- 예: `P20260001`
- 기존 코드는 그대로 이관
- 신규 등록 시 시스템 자동 제안, 관리자 수정 허용

#### 부서장 확인 포인트
- 기존 코드 체계를 최대한 유지할지
- 신규 등록부터는 형식을 통일할지
- 코드 자동생성을 허용할지

---

### 2-3. 날짜 입력 정책

#### 왜 먼저 정해야 하나
현재 엑셀은 날짜, 시간, 설명 텍스트, 미정 문구가 혼재되어 있어 자동 집계가 어렵다.
날짜 관련 입력 정책은 초기에 반드시 구조화해야 한다.

#### 기존 엑셀에서 보이는 방식
- 순수 날짜: `2026-03-10`
- 날짜+시간: `2025-11-14 15:00`
- 날짜+설명: `2024-07-10 (현장 제출)`
- 복수 날짜: `2024-09-25 (공고) / 2024-10-08 (재공고)`
- 미정/발표없음 성격의 텍스트 가능

#### 제안 목록
- 날짜 입력을 분리
  - `일자`
  - `시간`
  - `유형/방식`
  - `비고`
- 미정은 텍스트 대신
  - `미정 여부` 체크
  - `비고`에 설명

#### 부서장 확인 포인트
- 날짜만으로 충분한지, 시간까지 필요한지
- `현장 제출`, `온라인 제출`, `재공고` 같은 정보를 별도 필드로 뺄지
- 발표 미정 시 표현 방식을 어떻게 통일할지

---

### 2-4. 상태 전환 규칙

#### 왜 먼저 정해야 하나
상태값 이름만 정해서는 부족하다.
어떤 상태에서 어떤 상태로 이동할 수 있는지까지 정해야 저장 로직과 검증 로직을 만들 수 있다.

#### 기존 엑셀에서 보이는 방식
- 전환 규칙이 시스템적으로 강제된 흔적은 없음
- 운영상 다음과 같은 흐름이 암묵적으로 보임
  - 제안진행 → WIN / LOSS / DROP
  - WIN → 수행중(WIN)
  - 업무지원 → 완료

#### 제안 목록
- 기본 흐름
  - `제안준비 → 제안중 → 발표완료 → 수주/실주 → 수행중 → 완료`
- 예외 흐름
  - `제안중 → 중단`
  - `수행중 → 중단`

#### 부서장 확인 포인트
- `발표완료` 후 반드시 `수주/실주`를 거치게 할지
- `제안준비`를 별도로 둘지
- `협력참여` 사업도 같은 흐름을 따르는지

---

### 2-5. 대기인원 / 중복투입 기준

#### 왜 먼저 정해야 하나
가동률, 가득률, 대기인원, 월간 집계의 숫자를 일관되게 맞추려면
“누가 대기인지”, “누가 중복투입인지”를 분명히 정해야 한다.

#### 기존 엑셀에서 보이는 목록
- `Ⅱ.1.주간현황`
  - 비가동 = `대기 + 사업부장 + 지원`
- `인원별_투입(현재)`
  - `대기`
  - `휴직`
  - `퇴직`
  - `전배`
  - `사업부장`
  - `미투입`

#### 제안 목록
- 인력 상태 구분
  - `수행`
  - `제안`
  - `지원`
  - `대기`
  - `휴직`
  - `퇴직`
  - `전배`
  - `관리직`
- 대기인원 정의
  - **유효 재직자 중 해당 시점에 어떤 프로젝트에도 배정되지 않은 인력**
- 중복투입 규칙
  - 같은 기간 복수 프로젝트 배정 허용
  - 단, 배정률 합계로 관리

#### 부서장 확인 포인트
- `지원`을 비가동으로 볼지
- `사업부장/팀장`을 별도 관리직으로 분리할지
- `휴직/전배/퇴직`을 대기인원 집계에서 완전히 제외할지

## 3. MVP 범위 관련 사전확정 항목

### 3-1. MVP 화면 범위

#### 왜 먼저 정해야 하나
MVP 범위를 고정해야 개발 순서와 일정이 안정된다.
중간에 화면 범위가 계속 늘어나면 바이브코딩 작업도 급격히 흔들린다.

#### 기존 엑셀이 대신하던 화면 역할
- 업무수행현황 원장
- 프로젝트코드 관리
- 월별 가동현황
- 대기/제안인원 현황
- 제안 프로젝트 현황
- 이행 프로젝트 현황
- 인력 재직현황
- 인력별 투입현황
- 인원별 현재 투입현황

#### 현재 제안 목록
- **MVP 포함**
  - 로그인
  - 대시보드
  - 인력 관리
  - 프로젝트 관리
  - 인력 배정
  - 진행이력
  - 월간 현황판
  - 보고서 다운로드
  - 사용자/권한 관리(관리자용)
- **2차 이후**
  - 주간 현황판
  - 알림 센터
  - 월마감/스냅샷 전용 화면
  - 외부 접속

#### 부서장 확인 포인트
- 관리자용 사용자/권한 관리 화면을 MVP에 넣을지
- 월간 현황판 상세 수준을 어디까지 볼지
- 주간 현황판을 정말 2차로 미뤄도 되는지

---

### 3-2. MVP 보고서 범위

#### 왜 먼저 정해야 하나
이 프로젝트는 “입력 화면”보다 “보고용 결과물”이 중요하다.
따라서 어떤 엑셀을 먼저 내려받아야 하는지를 초기에 확정해야 한다.

#### 기존 KPI 엑셀에서 보이는 보고 성격 시트
- `Ⅱ.1.주간현황`
- `Ⅱ.2.월별가동현황`
- `Ⅱ.3.대기,제안인원`
- `Ⅱ.4.제안PRJ`
- `Ⅱ.5.이행PRJ`
- `인원별_투입(현재)`
- `인력_투입현황MM`

#### 제안 목록
- **MVP 보고서**
  - 월간 가동 현황
  - 대기 / 제안 / 수행 인원 현황
  - 제안 프로젝트 현황
  - 수행 프로젝트 현황
  - 인력별 월간 투입 현황
- **2차 이후**
  - 주간 현황판
  - PDF 출력
  - 화면 인쇄 최적화

#### 부서장 확인 포인트
- 1순위 보고서가 무엇인지
- 기존 KPI 엑셀과 비슷하게 갈지, 더 단순화할지
- 본부장 보고용과 실무 운영용을 분리할지

---

## 4. 운영 책임자 기준 항목

### 4-1. 마스터 데이터 관리 책임자

#### 왜 먼저 정해야 하나
실제로 누가 데이터를 유지보수하는지 정해지지 않으면,
시스템은 만들어도 운영 품질이 금방 떨어진다.

#### 현재 운영 가정
- 최초 사업 등록: 관리자 / 본부장
- 진행이력 입력: 관리자 / 본부장
- 추후 PM에게 이관 가능

#### 제안 목록
- MVP
  - 인력 마스터: 관리자
  - 조직/영업대표 마스터: 관리자
  - 프로젝트 마스터: 관리자, 본부장
  - 진행이력: 관리자, 본부장
- 2차
  - PM에게 프로젝트/진행이력 입력 권한 일부 이관

#### 부서장 확인 포인트
- 본부장이 계속 직접 입력할지
- PM에게 어느 시점에 이관할지
- 팀장에게 수정 권한을 줄지

---

### 4-2. 월마감 책임자와 기준

#### 왜 먼저 정해야 하나
월마감/스냅샷은 2차 기능이지만, 구조는 MVP 때부터 고려해야 한다.
과거 값 재현과 보고 숫자 고정이 필요한지 여부를 미리 확인해야 한다.

#### 기존 엑셀에서 보이는 방식
- 월별 집계 구조는 있으나
- “월마감 버튼”, “수정 잠금”, “스냅샷 승인자”는 명시적이지 않음

#### 제안 목록
- 실행자
  - MVP 문서 기준: 관리자
  - 추후: 본부장 이관 가능
- 기준
  - 매월 말일 기준 스냅샷 저장
  - 월마감 후 일반 사용자 수정 제한
- 기능 구현은 2차로 진행

#### 부서장 확인 포인트
- 실제로 월 기준 확정값이 필요한지
- 지난달 숫자가 나중에 바뀌면 안 되는지
- 스냅샷만으로 충분한지, 수정 잠금까지 필요한지
