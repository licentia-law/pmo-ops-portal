# 휴직·전배·퇴직 MM 정확 반영을 위한 인력 상태 이력 설계안

작성일: 2026-06-17  
대상 시스템: PMO 업무수행 관리시스템 / pmo-ops-portal  
대상 영역: 관리자 > 인력 관리, 인력 > 인력 투입 현황, KPI/보고 > 주간현황·월별가동현황

---

## 1. 문서 목적

본 문서는 PMO 업무수행 관리시스템에서 **휴직, 전배, 퇴직까지 재직 MM 및 가동률/가득률 계산에 정확히 반영**하기 위해 필요한 데이터 구조, 화면 항목, 계산 기준, API 구현 방향을 정리한 설계 문서이다.

현재 논의 중인 인력 관리 기본 항목은 다음과 같다.

```text
재직상태
MM 시작일
MM 종료일
연간 재직 MM
사용여부
```

이 구조는 단순한 연속 재직 기간을 관리하기에는 적합하지만, 다음과 같은 복합 케이스를 정확히 계산하기에는 부족하다.

```text
2026-01-01 ~ 2026-12-31 재직
2026-06-01 ~ 2026-07-31 휴직
2026-09-01 PMO1팀 → PMO2팀 전배
2026-11-15 퇴직
```

따라서 정확한 MM 계산을 위해서는 인력 마스터의 단일 상태값 외에 **기간 단위의 인력 상태 이력**이 필요하다.

---

## 2. 핵심 결론

휴직, 전배, 퇴직을 MM에 정확히 반영하려면 인력 마스터의 `재직상태` 단일값만으로는 부족하다.

반드시 아래 개념이 추가되어야 한다.

```text
인력별 상태 이력
= 상태 + 시작일 + 종료일 + MM 반영 비율 + 전배 전/후 소속 + 사유/메모
```

권장 구조는 다음과 같다.

```text
인력 마스터
- 인력의 기본정보와 현재 대표 상태를 관리
- 목록 화면은 단순하게 유지

인력 상태 이력
- 휴직, 전배, 퇴직, 복직 등 기간성 이벤트를 관리
- 재직 MM, 팀별 MM, 가동률/가득률 계산의 기준 데이터로 사용
```

## 3. 현재 항목만으로 가능한 것과 불가능한 것

| 항목 | 현재 구조로 가능 여부 | 설명 |
|---|---:|---|
| 연초부터 연말까지 정상 재직 | 가능 | MM 시작일~종료일만으로 계산 가능 |
| 중도 입사 또는 중도 합류 | 가능 | MM 시작일을 중도 날짜로 입력하면 계산 가능 |
| 중도 퇴직 | 가능 | MM 종료일을 퇴직일로 입력하면 계산 가능 |
| PMO 외부 전배 | 부분 가능 | MM 종료일로 처리 가능하지만 전배 이력은 남지 않음 |
| PMO 내부 팀 이동 | 불완전 | 전체 MM은 유지되지만 팀별 MM 배분이 정확하지 않음 |
| 중간 휴직 후 복직 | 불가능 | 단일 MM 시작일/종료일로 중간 공백 기간을 제외하기 어려움 |
| 휴직·복직 반복 | 불가능 | 여러 기간을 표현할 수 없음 |
| 퇴직 전 휴직 후 퇴직 | 불완전 | 상태 변화를 순서대로 기록하기 어려움 |
| 특정 기간만 MM 반영률 조정 | 불가능 | 0.5 MM, 부분 가용 등 비율 반영 불가 |

## 4. 추천 데이터 구조

### 4.1 인력 마스터 테이블

인력 마스터는 복잡한 상태 이력 전체를 담는 테이블이 아니라, 인력의 현재 기본정보와 대표 상태를 관리하는 테이블로 둔다.

권장 필드:

| 필드 | 설명 | 비고 |
|---|---|---|
| id | 인력 ID | PK |
| employee_no | 사번 | 화면 표시 및 검색 기준 |
| name | 성명 | 필수 |
| email | 이메일 | 선택 또는 준필수 |
| group_name | 본부 | 문자열 관리 가능 |
| team_name | 팀 | 문자열 관리 가능 |
| position | 직위 | 예: 이사, 책임, 수석, 선임 |
| role | 대표 역할 | 예: PM, PL, 개발, TA, QA |
| current_employment_status | 현재 재직상태 | 상태 이력 기준으로 계산 또는 동기화 |
| mm_start_date | MM 시작일 | 최초 산정 시작 기준 |
| mm_end_date | MM 종료일 | 최종 산정 종료 기준. 없으면 계속 재직 |
| annual_employment_mm | 연간 재직 MM | 직접 입력보다 계산값 권장 |
| is_active | 사용여부 | 사용/미사용 |
| memo | 비고 | 선택 |
| created_at | 생성일시 | 시스템 관리 |
| updated_at | 수정일시 | 시스템 관리 |

권장 명칭:

```text
계약 시작일 / 계약 종료일
→ MM 시작일 / MM 종료일
```

`계약 시작일`이라는 명칭은 고용계약, 프로젝트 계약, PMO 산정 기간을 혼동시킬 수 있다. 이 화면의 목적이 재직 MM 산정이라면 `MM 시작일`, `MM 종료일`이 더 명확하다.

---

### 4.2 인력 상태 이력 테이블

정확한 MM 계산을 위해 별도 이력 테이블을 둔다.

권장 테이블명 예시:

```text
personnel_status_histories
```

권장 필드:

| 필드 | 설명 | 필수 여부 | 비고 |
|---|---|---:|---|
| id | 상태 이력 ID | 필수 | PK |
| personnel_id | 대상 인력 ID | 필수 | personnel.id FK |
| status | 상태 | 필수 | active, leave, transferred, retired, waiting 등 |
| status_detail | 상태 상세 | 선택 | 내부전배, 외부전배, 복직 등 세부 구분 |
| start_date | 상태 시작일 | 필수 | MM 계산 시작 기준 |
| end_date | 상태 종료일 | 선택 | 비어 있으면 현재까지 지속 |
| mm_weight | MM 반영 비율 | 필수 | 정상 재직 1.0, 휴직 0.0 등 |
| from_group_name | 변경 전 본부 | 선택 | 전배 시 사용 |
| from_team_name | 변경 전 팀 | 선택 | 전배 시 사용 |
| to_group_name | 변경 후 본부 | 선택 | 전배 시 사용 |
| to_team_name | 변경 후 팀 | 선택 | 전배 시 사용 |
| reason | 사유 | 선택 | 휴직 사유, 전배 사유, 퇴직 사유 |
| memo | 메모 | 선택 | 상세 설명 |
| created_by | 등록자 | 선택 | 로그인 사용자 ID |
| created_at | 등록일시 | 필수 | 시스템 자동 |
| updated_by | 수정자 | 선택 | 로그인 사용자 ID |
| updated_at | 수정일시 | 필수 | 시스템 자동 |

## 5. 상태값 설계

### 5.1 기본 상태값

| 상태 | 내부값 예시 | 의미 | MM 반영 기본값 |
|---|---|---|---:|
| 재직 | active | 정상 재직 상태 | 1.0 |
| 휴직 | leave | 휴직 기간 | 0.0 |
| 전배 | transferred | 소속 변경 | 정책에 따라 1.0 또는 0.0 |
| 퇴직 | retired | 퇴직 이후 | 0.0 |
| 대기 | waiting | 프로젝트 미투입 상태 | 1.0 |

주의할 점은 `대기`는 재직 MM 기준에서는 보통 포함된다. 다만 가동률/가득률 계산에서는 미투입 또는 비가동으로 분류될 수 있다.

---

### 5.2 전배 유형 구분

전배는 반드시 세부 구분이 필요하다.

| 전배 유형 | 설명 | 전체 PMO MM | 팀별 MM |
|---|---|---:|---|
| 내부전배 | PMO본부 내부 팀 이동 | 유지 | 이동일 기준으로 팀별 배분 |
| 외부전배 | PMO본부 외부 조직으로 이동 | 제외 | 전배일 이후 제외 |

권장 내부값:

```text
transfer_internal
transfer_external
```

또는 `status = transferred`, `status_detail = internal/external` 구조로 관리한다.

## 6. 상태별 계산 기준

### 6.1 정상 재직

```text
status = active
mm_weight = 1.0
```

해당 기간은 재직 MM에 100% 반영한다.

---

### 6.2 휴직

```text
status = leave
mm_weight = 0.0
```

휴직 기간은 재직 MM에서 제외하는 것을 기본 정책으로 한다.

예시:

```text
2026-01-01 ~ 2026-12-31 재직
2026-06-01 ~ 2026-07-31 휴직
```

계산 방식:

```text
1월~5월: MM 반영
6월~7월: MM 제외
8월~12월: MM 반영
```

---

### 6.3 내부전배

```text
status = transferred
status_detail = internal
mm_weight = 1.0
```

PMO본부 전체 재직 MM은 유지한다. 다만 팀별 집계는 전배일 기준으로 나눠야 한다.

예시:

```text
2026-01-01 ~ 2026-08-31 PMO1팀
2026-09-01 ~ 2026-12-31 PMO2팀
```

계산 방식:

```text
PMO본부 전체 MM: 계속 반영
PMO1팀 MM: 1월~8월 반영
PMO2팀 MM: 9월~12월 반영
```

---

### 6.4 외부전배

```text
status = transferred
status_detail = external
mm_weight = 0.0
```

외부전배일 이후 PMO본부 재직 MM에서 제외한다.

예시:

```text
2026-01-01 ~ 2026-08-31 PMO본부
2026-09-01 ~ 외부본부 전배
```

계산 방식:

```text
1월~8월: PMO MM 반영
9월~12월: PMO MM 제외
```

---

### 6.5 퇴직

```text
status = retired
mm_weight = 0.0
```

퇴직일 이후 MM에서 제외한다.

정책 결정 필요:

```text
퇴직일 당일까지 반영할 것인지
퇴직일 전일까지 반영할 것인지
```

MVP 권장 정책:

```text
퇴직일 전일까지 MM 반영
퇴직일부터 MM 제외
```

다만 회사 내부 기준이 있다면 그 기준을 우선 적용한다.

## 7. MM 계산 방식

### 7.1 기본 계산 원칙

월별 재직 MM은 다음 기준으로 계산한다.

```text
월별 재직 MM = 해당 월의 상태별 유효 근무일 수 / 해당 월의 전체 근무일 수 × MM 반영 비율
```

근무일 수 계산 시 주말과 공휴일은 제외한다.

```text
근무일 = 평일 - 공휴일
```

---

### 7.2 연간 재직 MM

연간 재직 MM은 1월~12월 월별 재직 MM의 합계다.

```text
연간 재직 MM = 1월 MM + 2월 MM + ... + 12월 MM
```

예시:

| 월 | 상태 | MM |
|---|---|---:|
| 1월 | 재직 | 1.0 |
| 2월 | 재직 | 1.0 |
| 3월 | 재직 | 1.0 |
| 4월 | 재직 | 1.0 |
| 5월 | 재직 | 1.0 |
| 6월 | 휴직 | 0.0 |
| 7월 | 휴직 | 0.0 |
| 8월 | 재직 | 1.0 |
| 9월 | 재직 | 1.0 |
| 10월 | 재직 | 1.0 |
| 11월 | 퇴직 전 일부 근무 | 0.x |
| 12월 | 퇴직 | 0.0 |

## 9. API 설계 제안

### 9.1 조회 API

```http
GET /api/personnel
GET /api/personnel/{id}
GET /api/personnel/{id}/status-histories
GET /api/monthly-employment-mm?year=2026&personnelId={id}
```

### 9.2 등록/수정 API

```http
POST /api/personnel
PATCH /api/personnel/{id}
POST /api/personnel/{id}/status-histories
PATCH /api/personnel/{id}/status-histories/{historyId}
DELETE /api/personnel/{id}/status-histories/{historyId}
```

### 9.3 집계 API

```http
POST /api/personnel/{id}/recalculate-employment-mm
POST /api/monthly-employment-mm/recalculate?year=2026
GET /api/monthly-employment-mm?year=2026
```

## 10. 검증 규칙

상태 이력 등록/수정 시 다음 검증이 필요하다.

| 검증 항목 | 설명 |
|---|---|
| 시작일 필수 | 모든 이력은 시작일 필요 |
| 종료일 >= 시작일 | 종료일이 시작일보다 빠르면 저장 불가 |
| 기간 중복 방지 | 동일 인력의 같은 기간에 상충 상태가 중복되면 경고 또는 차단 |
| 전배 유형 필수 | 전배 상태 선택 시 내부/외부 전배 구분 필요 |
| 전배 후 소속 필수 | 전배 상태 선택 시 변경 후 본부/팀 필요 |
| 퇴직 이후 상태 등록 제한 | 퇴직 이후 새 이력은 복직 등 별도 정책 없으면 제한 |
| MM 반영 비율 범위 | 0.0 이상 1.0 이하 권장 |
| 사용중 인력만 집계 | is_active=false 인력은 기본 집계 제외 |

## 11. MVP 구현 범위 제안

### 11.1 MVP에 포함할 것

```text
인력 마스터 목록/상세
상태 이력 등록/수정
휴직 기간 MM 제외
외부전배 이후 MM 제외
내부전배 시 팀별 MM 이동
퇴직일 이후 MM 제외
월별 재직 MM 재계산
```

### 11.2 MVP에서 제외 가능

## 14. 최종 판단

MVP에서 인력 관리 페이지의 목록 컬럼은 단순하게 유지하는 것이 맞다.

다만 휴직, 전배, 퇴직까지 MM에 정확히 반영하려면 다음 구조가 반드시 필요하다.

```text
인력 마스터: 현재 상태와 기본 정보 관리
인력 상태 이력: 기간별 상태 변경 관리
월별 재직 MM: 상태 이력 기반 계산 결과 저장
```

따라서 최종 권장안은 다음과 같다.

```text
관리자 > 인력 관리
- 인력 목록: 기본정보 + 현재상태 + 연간 재직 MM 표시
- 인력 상세: 상태 이력 관리
- MM 계산: 상태 이력 기반 백엔드 집계
```

이 구조를 적용하면 휴직, 내부전배, 외부전배, 퇴직을 모두 기간 단위로 정확히 계산할 수 있고, 이후 주간현황/월별가동현황의 기준 인원과 가동률 계산에도 일관되게 반영할 수 있다.
